# Анализ и диагностика индексов в PostgreSQL

## Превью / аннотация / краткое содержание

Занятие посвящено анализу влияния индексов на выполнение SQL-запросов и диагностике состояний, снижающих их эффективность.
 Центральная проблема связана с тем, что наличие индекса не гарантирует ускорения запроса без корректного порядка полей, актуальной статистики, обслуживания таблиц и понимания плана выполнения.
 Материал раскрывает связь между индексами, EXPLAIN ANALYZE, BUFFERS, Index Scan, Index Only Scan, мёртвыми строками, VACUUM, ANALYZE и работой оптимизатора.
 Практическая логика занятия строится вокруг проверки реального эффекта индекса через план запроса и системные статистики PostgreSQL.

## Основная часть

## Индекс как структура доступа к данным

Индекс представляет собой вспомогательную структуру данных, предназначенную для ускорения поиска строк по заданному признаку. В учебной модели он объясняется как «оглавление» для таблицы: вместо последовательного просмотра всех строк СУБД получает возможность перейти к нужной области данных через упорядоченную структуру. Наиболее типичная модель для реляционных СУБД связана с B-tree, где значения организованы иерархически: от корневого узла к промежуточным и листовым элементам. Такой способ доступа особенно эффективен при точечном поиске и диапазонных условиях. Индекс ускоряет чтение, но создаёт дополнительную стоимость хранения и сопровождения при изменении данных.

### Индекс и последовательное сканирование

Последовательное сканирование таблицы означает чтение всех или почти всех строк независимо от того, сколько записей нужно вернуть в результате. При отсутствии подходящего индекса запрос по неиндексированному полю вынужден выполнять Seq Scan или Parallel Seq Scan, что приводит к значительному числу прочитанных буферов и росту времени выполнения. Индекс позволяет изменить физический способ доступа: СУБД ищет значения по индексной структуре и затем извлекает соответствующие строки. Однако сам факт наличия индекса не означает его автоматического использования. Оптимизатор выбирает индекс только тогда, когда оценивает его дешевле полного чтения таблицы.

## Диагностика эффекта от индекса

Оценка индекса выполняется через сравнение плана запроса до и после его создания. Для этого используется EXPLAIN ANALYZE, позволяющий увидеть фактическое время выполнения, количество строк, тип доступа и операции чтения. Параметр BUFFERS показывает, сколько страниц было прочитано из shared buffers или с диска, поэтому он связывает логический план с физической стоимостью доступа к данным. В практическом примере запрос по имени пассажира сначала выполняется через Parallel Seq Scan, удаляя фильтром большое количество строк. После создания индекса план меняется на индексный доступ, а время выполнения и объём чтения резко сокращаются.

### План запроса как критерий полезности индекса

Полезность индекса определяется не его существованием, а изменением плана выполнения и фактических метрик. Если после создания индекса запрос продолжает читать таблицу полностью, значит индекс либо не соответствует условию, либо условие имеет низкую селективность, либо статистика не позволяет оптимизатору выбрать индексный план. Ключевые признаки положительного эффекта: исчезновение полного сканирования большой таблицы, снижение Execution Time, уменьшение числа прочитанных буферов, переход к Index Scan или Index Only Scan. Поэтому диагностика индекса должна опираться на измерения, а не на предположение о том, что индекс всегда ускоряет запрос.

## Составной индекс и порядок полей

Составной индекс строится по нескольким столбцам и эффективен только при корректном порядке ключей. В PostgreSQL порядок полей определяет, какие условия могут быть использованы для быстрого доступа. Если индекс создан как (ticket_no, passenger_name), а запрос фильтрует только по passenger_name, использование индекса может быть ограниченным или невозможным. Если индекс создан как (passenger_name, ticket_no), то условие по passenger_name соответствует первому ключу индекса и становится пригодным для эффективного поиска. Поэтому составной индекс проектируется не по списку доступных полей, а по реальному порядку фильтрации в запросах.

### Поля отбора и покрытие запроса

Порядок полей индекса должен соответствовать тому, как данные чаще всего отбираются в запросах. Поля, используемые для точного отбора и имеющие высокую селективность, обычно размещаются раньше. Дополнительные поля могут включаться в индекс, чтобы покрыть запрос и уменьшить обращения к основной таблице. В практическом примере индекс по passenger_name и ticket_no позволяет не только отобрать строки по имени пассажира, но и вернуть номер билета из самой индексной структуры. Это создаёт предпосылку для Index Only Scan, при котором запрос может быть выполнен без чтения heap-таблицы.

## Index Scan и Index Only Scan

Index Scan использует индекс для поиска подходящих строк, но может обращаться к основной таблице для получения недостающих столбцов или проверки видимости строк. Index Only Scan является более экономичным вариантом: запрос получает все необходимые данные из индекса и не читает heap-таблицу, если это позволяет карта видимости. Такой план особенно эффективен для покрывающих индексов, где индекс содержит все поля, используемые в выборке и фильтрации. В учебном примере переход к Index Only Scan сопровождается резким сокращением времени выполнения. Однако этот эффект зависит не только от состава индекса, но и от состояния таблицы после изменений.

### Visibility map и обращение к основной таблице

В PostgreSQL Index Only Scan возможен только при наличии достоверной информации о видимости строк. Для этого используется visibility map, показывающая, какие страницы содержат строки, видимые всем активным транзакциям. Если карта видимости неактуальна, СУБД вынуждена обращаться к heap-таблице для проверки строк, даже если все нужные поля есть в индексе. Поэтому после массовых изменений таблицы может потребоваться VACUUM, чтобы обновить служебную информацию и восстановить эффективность индексного чтения. Практический вывод состоит в том, что покрывающий индекс и обслуживание таблицы должны рассматриваться совместно.

## Обновление строк и механизм MVCC

PostgreSQL реализует многоверсионность данных через MVCC, поэтому обновление строки физически не заменяет запись на месте. Вместо этого старая версия строки становится неактуальной, а новая версия добавляется как отдельная запись. Такая модель обеспечивает конкурентный доступ и согласованность транзакций, но приводит к накоплению мёртвых строк. На уровне логики пользователь видит одну актуальную запись, однако физически таблица может содержать старые версии, которые больше не участвуют в результате запроса. Эти версии продолжают занимать место и могут увеличивать объём чтения, если не выполняется обслуживание таблицы.

### Мёртвые строки и увеличение чтения

Мёртвые строки влияют на производительность, поскольку СУБД приходится учитывать дополнительные физические записи и страницы. После серии обновлений таблица может стать больше физически, хотя логический объём данных почти не изменился. При выполнении запросов увеличивается число прочитанных буферов, а операции сканирования и агрегации становятся дороже. В плане выполнения это проявляется через рост Buffers, сохранение или усиление Parallel Seq Scan, а также увеличение времени чтения. Поэтому диагностика индексов невозможна без анализа состояния таблицы: медленное чтение может быть вызвано не отсутствием индекса, а накопленным bloat.

## VACUUM и обслуживание таблиц

VACUUM в PostgreSQL очищает неактуальные версии строк и обновляет служебную информацию, необходимую для эффективного чтения. Он не всегда немедленно уменьшает физический размер файла таблицы, но делает пространство пригодным для повторного использования и обновляет visibility map. VACUUM FULL выполняет более радикальное перестроение таблицы, физически освобождая место, но является более тяжёлой операцией и может требовать блокировок. Autovacuum автоматизирует обслуживание, однако при интенсивных обновлениях или отключении autovacuum таблица быстро накапливает мёртвые строки. Поэтому регулярное обслуживание является частью производительности, а не отдельной административной формальностью.

### VACUUM, ANALYZE и состояние индексного доступа

VACUUM и ANALYZE решают разные, но взаимосвязанные задачи. VACUUM обслуживает физическое состояние таблицы и видимость строк, а ANALYZE обновляет статистику, используемую оптимизатором. После выполнения VACUUM ANALYZE индексный доступ может стать эффективнее не потому, что сам индекс изменился, а потому что СУБД получила актуальную информацию о видимости и распределении данных. В практическом примере после обновления карты видимости Index Only Scan выполняется без обращений к heap, что подтверждается нулевым или минимальным количеством Heap Fetches. Следовательно, индекс требует сопровождения всей таблицы, в которой он используется.

## Статистика таблиц и индексов

Статистика в PostgreSQL используется оптимизатором для оценки количества строк, селективности условий, размеров таблиц и стоимости альтернативных планов. Без статистики оптимизатор вынужден опираться на общие предположения, которые могут не соответствовать реальному распределению данных. Например, для пустой или недавно созданной таблицы СУБД может ожидать условное количество страниц и строк, поскольку фактические сведения ещё не собраны. После загрузки данных или массовых изменений без ANALYZE планировщик может продолжать строить план на устаревшей картине. Поэтому актуальность статистики напрямую влияет на выбор индекса, JOIN и способа сканирования.

### Оценка кардинальности и выбор плана

Кардинальность показывает, сколько строк ожидается после фильтрации, соединения или агрегации. Если статистика устарела, оптимизатор может недооценить или переоценить объём данных и выбрать неправильный план. Например, он может ожидать небольшую таблицу и выбрать Nested Loop, хотя фактически требуется обработка крупного набора. Или наоборот, отказаться от индекса, потому что статистика не отражает высокой селективности условия. В диагностике это видно через расхождение между rows в оценке и фактическим числом строк в EXPLAIN ANALYZE. Обновление статистики через ANALYZE является базовым шагом перед выводами о необходимости новых индексов.

## Системные представления для диагностики

PostgreSQL предоставляет системные представления, позволяющие оценивать состояние таблиц и индексов. pg_stat_all_tables содержит сведения о числе мёртвых строк, изменениях после последнего анализа и другой эксплуатационной статистике. Поле n_dead_tup помогает выявлять накопление мёртвых строк, а n_mod_since_analyze показывает количество изменений, произошедших после последнего сбора статистики. Эти показатели позволяют понять, почему план запроса мог стать хуже: таблица физически разрослась, карта видимости устарела или статистика больше не соответствует данным. Аналогичные диагностические инструменты существуют и в других СУБД, например для анализа физического состояния индексов в MS SQL Server.

### Диагностика до создания нового индекса

Перед добавлением нового индекса необходимо проверить, не связана ли проблема с уже существующими индексами, устаревшей статистикой или отсутствием обслуживания. Новый индекс может ускорить конкретный запрос, но одновременно увеличить стоимость вставок, обновлений и удаления. Поэтому последовательность диагностики должна включать анализ плана, проверку BUFFERS, оценку мёртвых строк, актуальность статистики и соответствие порядка полей индекса реальным условиям отбора. Только после этого можно обоснованно решать, нужен ли новый индекс, требуется ли перестроение существующего или достаточно выполнить VACUUM ANALYZE.

## Индексирование как инженерное решение

Индексирование является инженерным компромиссом между скоростью чтения и стоимостью сопровождения данных. Индекс ускоряет запросы отбора, сортировки и соединения, но требует дискового пространства и обновляется при изменении строк. Чем больше индексов создано на таблице, тем дороже становятся операции записи. Поэтому индекс не должен рассматриваться как универсальное средство ускорения. Его проектирование должно учитывать профиль запросов, частоту чтения и записи, порядок условий, покрытие выборки, селективность и стоимость обслуживания.

### Ограничения универсальных рекомендаций

Рекомендация «добавить индекс» корректна только после подтверждения через план выполнения и статистики. Если запрос читает большую часть таблицы, индекс может быть бесполезен. Если индекс создан с неправильным порядком полей, он может не использоваться для основного условия отбора. Если таблица содержит много мёртвых строк или устаревшую статистику, проблема может сохраняться даже при наличии подходящего индекса. Поэтому эффективная диагностика индексов требует связать четыре уровня: текст запроса, план выполнения, физическое состояние таблицы и статистику, на основе которой работает оптимизатор.

## Итоговые выводы

Индекс ускоряет запрос только при соответствии ключей индекса условиям отбора, селективности данных и фактическому плану выполнения.

Составные индексы требуют корректного порядка полей; первыми должны располагаться столбцы, чаще всего используемые для отбора.

Index Only Scan эффективен при покрытии запроса индексом и актуальной visibility map, поддерживаемой через VACUUM.

Мёртвые строки, bloat и устаревшая статистика могут ухудшать планы запросов даже при наличии подходящих индексов.

Главный результат освоения занятия состоит в умении диагностировать индексы через план выполнения, буферы, статистику таблиц и состояние обслуживания, а не через механическое добавление новых индексов.

## 00:00–03:00. Начало занятия и постановка темы

Занятие посвящено анализу и диагностике индексов в PostgreSQL. Эта тема продолжает блок о производительности баз данных и чтении планов выполнения запросов. Если ранее рассматривалось, как СУБД выполняет запрос и какие операции появляются в плане, то здесь основной акцент переносится на индексы: когда они действительно помогают, почему иногда не используются и как проверить их реальный эффект.

Главная идея занятия состоит в том, что индекс нельзя рассматривать как универсальное средство ускорения. Само наличие индекса на таблице ещё не означает, что запрос станет быстрее. Оптимизатор может отказаться от индекса, если считает полное сканирование таблицы дешевле, если статистика устарела или если условие запроса не соответствует структуре индекса.

Поэтому работа с индексами должна строиться через диагностику. Нужно смотреть план выполнения, фактическое время, количество строк, буферы, обращения к heap, состояние таблицы, количество мёртвых строк и актуальность статистики. Только после этого можно делать вывод, нужен ли новый индекс, правильно ли построен существующий и требуется ли обслуживание таблицы.

## 03:00–06:00. Индекс как структура быстрого доступа

Индекс объясняется как вспомогательная структура данных, которая помогает СУБД быстрее находить строки в таблице. По смыслу он похож на оглавление или указатель в книге: чтобы найти нужный фрагмент, не нужно читать всю книгу подряд, можно перейти к нужному месту через заранее подготовленную структуру.

В PostgreSQL наиболее распространённым типом индекса является B-tree. Такая структура организует значения в виде дерева: есть корневой уровень, промежуточные узлы и листовые страницы. Поиск проходит не через все строки таблицы, а через последовательное сужение области поиска.

Индекс особенно полезен при точечном поиске, диапазонном поиске, сортировке и соединении таблиц по ключу. Но индекс не бесплатен. Он занимает место на диске и должен обновляться при вставке, обновлении и удалении строк. Поэтому большое количество индексов может ускорять чтение, но замедлять запись.

## 06:00–09:00. Почему индекс не всегда используется

Один из центральных моментов занятия — индекс не обязан использоваться, даже если он существует. Решение принимает оптимизатор запросов. Он сравнивает разные варианты плана и выбирает тот, который считает дешевле.

Если запрос возвращает большую часть таблицы, индекс может оказаться менее выгодным, чем последовательное сканирование. При индексном доступе СУБД сначала обращается к индексу, затем часто идёт в основную таблицу за строками. Если таких строк слишком много, множественные обращения могут быть дороже, чем одно последовательное чтение таблицы.

Индекс также может не использоваться, если условие написано в неудобной форме. Например, если к индексируемому столбцу применяется функция, если происходит неявное приведение типов или если фильтр не соответствует первому столбцу составного индекса. Поэтому анализ индекса всегда начинается с плана выполнения.

## 09:00–12:00. Диагностика через EXPLAIN ANALYZE

Для проверки влияния индекса используется EXPLAIN ANALYZE. Эта команда выполняет запрос и показывает фактический план: какие операции были выбраны, сколько строк обработано, сколько времени заняла каждая часть плана и какой способ доступа применялся.

Если добавить параметр BUFFERS, можно увидеть, сколько страниц было прочитано из буферов PostgreSQL и насколько запрос связан с физическим чтением данных. Это помогает отличить чисто логическую проблему запроса от проблемы большого объёма чтения.

Примерная форма анализа:

EXPLAIN (ANALYZE, BUFFERS)

SELECT ...

В диагностике индексов особенно важны операции Seq Scan, Parallel Seq Scan, Index Scan, Index Only Scan, Bitmap Index Scan и Bitmap Heap Scan. По ним можно понять, читает ли база таблицу целиком или использует индексную структуру.

## 12:00–15:00. Запрос без подходящего индекса

В практической части демонстрируется запрос, который ищет данные по определённому полю, например по имени пассажира. До создания индекса PostgreSQL вынужден читать большую таблицу и проверять строки фильтром.

В плане такого запроса появляется Parallel Seq Scan. Это означает, что таблица читается последовательно, но с использованием параллельных воркеров. База пытается ускорить полный проход за счёт параллелизма, но принципиально всё равно обрабатывает большой объём данных.

В плане видно, что значительная часть строк отбрасывается фильтром. Это важный диагностический признак: запрос возвращает мало данных, но для этого просматривает много строк. В такой ситуации индекс потенциально может дать заметный эффект.

Однако вывод о необходимости индекса делается не по ощущению, а по плану: если таблица большая, фильтр селективный, а запрос часто используется, индекс по полю отбора становится обоснованным кандидатом.

## 15:00–18:00. Создание простого индекса

После анализа запроса создаётся индекс по столбцу, который используется в условии фильтрации. Например, если запрос ищет пассажира по имени, создаётся индекс по passenger_name.

Общая форма:

CREATE INDEX имя_индекса

ON имя_таблицы (имя_столбца);

После создания индекса тот же запрос выполняется повторно через EXPLAIN ANALYZE. Важно не просто создать индекс, а сравнить план до и после. Если всё сработало правильно, вместо полного сканирования таблицы появляется индексный доступ.

Это показывает практическую логику работы: сначала фиксируется проблема, затем создаётся гипотеза, после этого вносится изменение и снова измеряется результат. Индекс считается полезным только тогда, когда фактические метрики стали лучше.

## 18:00–21:00. Изменение плана после создания индекса

После создания индекса план запроса меняется. Вместо Parallel Seq Scan может появиться Index Scan или другой индексный оператор. Это означает, что PostgreSQL использует индекс для поиска подходящих строк.

Время выполнения снижается, потому что база больше не читает таблицу целиком. Она проходит по индексной структуре, находит нужные значения и извлекает соответствующие строки. Также уменьшается количество прочитанных буферов, что особенно важно для больших таблиц.

Но сам факт появления Index Scan ещё не всегда означает идеальный план. Нужно смотреть, сколько строк реально возвращается, сколько обращений к таблице выполняется, какие буферы читаются и насколько результат стабилен при повторном запуске.

Основной вывод этого блока: индекс должен подтверждаться планом и измерениями. Если план не изменился или время почти не улучшилось, значит проблема может быть не в отсутствии индекса.

## 21:00–24:00. Составной индекс

Далее рассматриваются составные индексы. Составной индекс создаётся не по одному столбцу, а сразу по нескольким. Например:

CREATE INDEX имя_индекса

ON имя_таблицы (столбец_1, столбец_2);

Такие индексы полезны, когда запросы часто фильтруют данные по нескольким полям одновременно. Но порядок полей в составном индексе имеет принципиальное значение. Индекс (ticket_no, passenger_name) и индекс (passenger_name, ticket_no) — это не одно и то же.

Если запрос фильтрует только по passenger_name, а в индексе первым стоит ticket_no, такой индекс может не помочь. PostgreSQL эффективно использует левую часть составного индекса. Поэтому первым в индексе обычно должно стоять поле, которое чаще используется для отбора и сильнее сужает результат.

## 24:00–27:00. Порядок полей в составном индексе

Порядок полей в индексе определяется не тем, как удобно перечислить колонки, а тем, как реально пишутся запросы. Если основные запросы ищут пассажира по имени и затем уточняют номер билета, логичнее поставить passenger_name первым, а ticket_no вторым.

Если же запросы почти всегда начинаются с номера билета, а имя используется только как дополнительное условие, тогда порядок может быть обратным. Поэтому проектирование составного индекса требует анализа частых запросов.

Ошибочный порядок полей может привести к тому, что индекс формально существует, но не используется для нужного сценария. Это одна из распространённых причин, почему разработчик видит индекс в схеме, но план всё равно показывает полное сканирование или менее эффективный доступ.

Практический принцип: составной индекс строится под конкретные условия фильтрации, сортировки и соединения. Его нельзя проектировать без понимания реальной нагрузки.

## 27:00–30:00. Покрывающий индекс

Следующий шаг — идея покрывающего индекса. Индекс может не только помогать найти строки, но и содержать все данные, которые нужны запросу. В таком случае СУБД может получить результат из индекса, не обращаясь к основной таблице.

Например, если запрос фильтрует по passenger_name и возвращает ticket_no, индекс по (passenger_name, ticket_no) может покрыть этот запрос. То есть оба нужных поля находятся в индексной структуре.

Это создаёт возможность для Index Only Scan. Такой план обычно быстрее обычного Index Scan, потому что не требует дополнительного чтения строк из heap-таблицы. Но для этого должны выполняться дополнительные условия, связанные с видимостью строк.

Покрывающий индекс особенно полезен для частых запросов чтения, которые возвращают небольшой набор полей. Но он тоже увеличивает размер индекса и стоимость обновлений, поэтому должен использоваться осознанно.

## 30:00–33:00. Index Scan и Index Only Scan

Index Scan означает, что PostgreSQL использует индекс для поиска строк, но затем обращается к основной таблице за данными или для проверки видимости строк. Это обычный и часто полезный способ доступа.

Index Only Scan означает, что база может выполнить запрос только по индексу. Если все нужные столбцы есть в индексе и информация о видимости строк актуальна, чтение heap-таблицы не требуется.

Разница между этими операциями важна для производительности. При Index Scan база может выполнять много дополнительных обращений к таблице. При Index Only Scan эти обращения сокращаются или полностью исчезают.

В плане выполнения для Index Only Scan нужно смотреть показатель Heap Fetches. Если он равен нулю или очень мал, значит PostgreSQL действительно получил данные почти полностью из индекса. Если Heap Fetches большой, эффективность такого плана снижается.

## 33:00–36:00. Visibility map и проверка видимости строк

Чтобы выполнить настоящий Index Only Scan, PostgreSQL должен знать, что строки на страницах таблицы видимы всем текущим транзакциям. Для этого используется visibility map. Это служебная структура, которая показывает, какие страницы таблицы можно считать полностью видимыми.

Если visibility map актуальна, PostgreSQL может доверять индексу и не обращаться к heap для проверки каждой строки. Если карта видимости неактуальна, база вынуждена идти в основную таблицу, даже если все нужные поля есть в индексе.

Поэтому покрывающий индекс сам по себе не гарантирует идеальный Index Only Scan. Нужна ещё корректная служебная информация о состоянии таблицы. Она обновляется в процессе обслуживания, прежде всего через VACUUM.

Это важная связь между индексами и обслуживанием таблиц. Иногда запрос не ускоряется не потому, что индекс плохой, а потому что таблица давно не обслуживалась.

## 36:00–39:00. MVCC и обновление строк

PostgreSQL использует механизм MVCC — многоверсионное управление конкурентным доступом. Его смысл в том, что разные транзакции могут видеть разные версии строк, не блокируя друг друга без необходимости.

При обновлении строки PostgreSQL не перезаписывает её физически на месте. Старая версия строки остаётся в таблице, а новая версия добавляется отдельно. С точки зрения пользователя запись изменилась, но физически в таблице появились две версии: старая неактуальная и новая актуальная.

Старая версия со временем становится мёртвой строкой, или dead tuple. Она больше не нужна для обычного чтения, но продолжает занимать место до тех пор, пока не будет очищена.

Такой механизм обеспечивает надёжную конкурентную работу, но создаёт необходимость регулярного обслуживания таблиц. Без обслуживания таблица физически разрастается, даже если логическое количество актуальных строк почти не меняется.

## 39:00–42:00. Мёртвые строки и bloat

Мёртвые строки увеличивают физический объём таблицы. Если таблица активно обновляется или из неё удаляются данные, dead tuples накапливаются. В результате база может читать больше страниц, чем требуется для актуальных данных.

Это состояние часто называют bloat. Таблица выглядит раздутой: логически данных может быть немного, но физически она занимает больше места и требует больше чтений. Индексы тоже могут разрастаться, потому что они содержат ссылки на версии строк.

В плане запроса это проявляется через рост количества буферов, увеличение времени выполнения и ухудшение эффективности сканирования. Иногда запрос кажется медленным из-за отсутствия индекса, но реальная причина — в плохом физическом состоянии таблицы.

Поэтому диагностика индексов должна включать не только анализ индекса, но и анализ таблицы: сколько в ней мёртвых строк, когда выполнялся VACUUM, насколько актуальна статистика.

## 42:00–45:00. Демонстрация влияния обновлений

В практической части показывается, что после массовых обновлений состояние таблицы меняется. Даже если запрос и индекс остаются прежними, план и фактические показатели могут ухудшиться из-за появления мёртвых строк и необходимости дополнительных проверок.

Например, Index Only Scan может начать выполнять больше Heap Fetches. Это означает, что PostgreSQL уже не может полностью доверять индексной структуре и вынужден обращаться к основной таблице для проверки видимости строк.

Количество прочитанных буферов может увеличиться. Время выполнения тоже может вырасти. При этом разработчик может ошибочно подумать, что индекс «сломался» или перестал работать, хотя проблема связана с MVCC и обслуживанием таблицы.

Этот пример показывает, что производительность индексов зависит не только от их структуры, но и от текущего физического состояния данных.

## 45:00–48:00. VACUUM

VACUUM — это механизм обслуживания таблиц в PostgreSQL. Он очищает мёртвые строки, освобождает пространство для повторного использования и обновляет служебные структуры, связанные с видимостью строк.

Обычный VACUUM не всегда уменьшает размер файла таблицы на диске. Он скорее помечает место внутри таблицы как пригодное для повторного использования. Это важно: после VACUUM таблица может занимать столько же места физически, но новые строки смогут использовать освобождённое пространство.

Для производительности запросов VACUUM важен ещё и потому, что он обновляет visibility map. Благодаря этому Index Only Scan может снова выполняться без лишних обращений к heap.

Таким образом, VACUUM влияет не только на освобождение места, но и на эффективность индексного доступа. Особенно это заметно на таблицах с частыми обновлениями и удалениями.

## 48:00–51:00. VACUUM FULL

VACUUM FULL отличается от обычного VACUUM. Он физически перестраивает таблицу и реально возвращает место операционной системе. То есть после VACUUM FULL размер таблицы на диске может уменьшиться.

Но эта операция тяжелее. Она требует больше времени, может создавать значительную нагрузку и обычно блокирует таблицу на время выполнения. Поэтому её нельзя применять как обычную регулярную операцию для всех таблиц подряд.

VACUUM FULL уместен, когда таблица сильно раздута и нужно физически вернуть место. Например, после массового удаления большого количества данных. Но в повседневной эксплуатации основную роль играет обычный VACUUM и autovacuum.

Практический вывод: VACUUM и VACUUM FULL решают разные задачи. Первый поддерживает рабочее состояние таблицы, второй выполняет более радикальную физическую очистку.

## 51:00–54:00. ANALYZE и статистика

ANALYZE собирает статистику по таблице. Эта статистика нужна оптимизатору, чтобы оценивать количество строк, селективность условий, распределение значений и стоимость разных планов выполнения.

Если статистика устарела, оптимизатор может ошибаться. Например, он может считать, что условие вернёт мало строк, хотя фактически оно возвращает много. Или наоборот, может отказаться от индекса, потому что неправильно оценивает селективность.

После массовой загрузки, обновления или удаления данных полезно выполнять ANALYZE, чтобы оптимизатор получил актуальную картину. Часто используется комбинированная команда:

VACUUM ANALYZE имя_таблицы;

Она одновременно обслуживает физическое состояние таблицы и обновляет статистику. Это базовый инструмент восстановления корректного планирования запросов.

## 54:00–57:00. Autovacuum

В PostgreSQL есть механизм autovacuum. Он автоматически запускает обслуживание таблиц: очищает мёртвые строки и обновляет статистику. Это необходимо, потому что вручную обслуживать все активно изменяемые таблицы неудобно и рискованно.

Autovacuum работает по порогам. Когда в таблице накапливается определённое количество изменений или мёртвых строк, PostgreSQL может запустить автоматический VACUUM или ANALYZE.

Но autovacuum не всегда успевает за нагрузкой. Если таблица очень активно обновляется, если настройки слишком мягкие или если autovacuum отключён, мёртвые строки могут накапливаться быстрее, чем система их очищает.

Поэтому в диагностике производительности нужно проверять, работает ли autovacuum и достаточно ли он агрессивен для конкретной нагрузки. Иногда проблема индексов на самом деле является проблемой обслуживания.

## 57:00–60:00. Системные представления PostgreSQL

Для диагностики PostgreSQL предоставляет системные представления. Одно из важных представлений — pg_stat_all_tables. Оно содержит статистику по таблицам: количество последовательных сканирований, индексных сканирований, вставок, обновлений, удалений, мёртвых строк и изменений после последнего анализа.

Особенно важны поля n_dead_tup и n_mod_since_analyze. n_dead_tup показывает примерное количество мёртвых строк. Если значение большое, таблица нуждается в обслуживании. n_mod_since_analyze показывает, сколько изменений произошло после последнего сбора статистики.

По этим данным можно понять, почему оптимизатор выбирает странный план. Если статистика устарела, он может неправильно оценивать количество строк. Если мёртвых строк много, физическое чтение может быть дороже, чем ожидается.

Такие представления позволяют диагностировать проблему до создания новых индексов. Иногда достаточно выполнить обслуживание, а не добавлять ещё одну индексную структуру.

## 60:00–63:00. Диагностика состояния индекса и таблицы

Диагностика индексов должна включать несколько шагов. Сначала нужно посмотреть план запроса через EXPLAIN ANALYZE. Затем нужно проверить, какой тип доступа используется: полное сканирование, индексное сканирование, bitmap-сканирование или Index Only Scan.

После этого нужно посмотреть буферы. Если запрос читает слишком много страниц, проблема может быть в отсутствии индекса, низкой селективности, большом количестве мёртвых строк или плохом физическом состоянии таблицы.

Затем проверяется статистика таблицы: когда выполнялся ANALYZE, сколько было изменений после него, сколько накопилось dead tuples. Если эти показатели плохие, сначала выполняется VACUUM ANALYZE, а уже потом снова проверяется план.

Такой порядок защищает от ошибки, когда разработчик добавляет новые индексы, хотя проблема была в устаревшей статистике или неочищенной таблице.

## 63:00–66:00. Когда нужен новый индекс

Новый индекс нужен тогда, когда есть частый или важный запрос, который выбирает небольшую часть данных, но вынужден читать слишком много строк. При этом условие запроса должно быть достаточно селективным, а индекс должен соответствовать этому условию.

Если запрос фильтрует по одному полю, может быть достаточно простого индекса. Если он часто фильтрует по нескольким полям, нужен составной индекс с правильным порядком колонок. Если запрос возвращает только поля, которые можно поместить в индекс, можно рассмотреть покрывающий индекс.

Но новый индекс добавляется только после проверки. Нужно посмотреть, изменился ли план, уменьшилось ли время выполнения, сократилось ли число буферов и не ухудшатся ли критично операции записи.

Индексирование — это компромисс. Оно ускоряет чтение, но усложняет вставку, обновление и удаление. Поэтому индекс должен решать конкретную измеренную проблему.

## 66:00–69:00. Когда индекс не решает проблему

Бывают ситуации, когда добавление индекса не помогает. Например, если запрос возвращает большую часть таблицы, оптимизатор может всё равно выбрать последовательное сканирование. Это нормально, потому что индексный доступ к большому объёму строк может быть дороже.

Индекс также не поможет, если проблема в тяжёлой сортировке, агрегации, JOIN без предварительного уменьшения данных или в неудачной архитектуре запроса. В таких случаях нужно переписывать SQL, менять порядок операций, использовать предварительную агрегацию или пересматривать модель данных.

Если таблица раздута мёртвыми строками, новый индекс может дать слабый эффект, потому что физическое состояние таблицы остаётся плохим. В этом случае сначала требуется обслуживание.

Если статистика устарела, оптимизатор может не выбрать даже хороший индекс. Поэтому перед выводами о бесполезности индекса нужно выполнить ANALYZE и повторить проверку плана.

## 69:00–72:00. Типичные ошибки при работе с индексами

Первая типичная ошибка — создавать индексы «на всякий случай». Это приводит к росту размера базы и замедлению операций записи. Каждый индекс нужно поддерживать при изменении данных.

Вторая ошибка — не учитывать порядок полей в составном индексе. Если первый столбец индекса не используется в условии, индекс может быть малоэффективен для нужного запроса.

Третья ошибка — игнорировать VACUUM и ANALYZE. Индекс может быть правильным, но таблица из-за dead tuples и устаревшей статистики всё равно будет работать плохо.

Четвёртая ошибка — смотреть только на наличие индекса, а не на план. Важно не то, есть ли индекс в схеме, а то, используется ли он фактически и улучшает ли реальные метрики запроса.

## 72:00–75:00. Методика анализа индекса

Практическая методика анализа индекса строится последовательно. Сначала выбирается конкретный медленный запрос. Затем строится план через EXPLAIN ANALYZE, желательно с BUFFERS. После этого фиксируется текущий способ доступа и фактические показатели.

Далее проверяется, есть ли подходящий индекс и соответствует ли он условиям запроса. Для составного индекса анализируется порядок колонок. Для покрывающего индекса проверяется, может ли запрос перейти к Index Only Scan.

После этого оценивается состояние таблицы: dead tuples, изменения после последнего анализа, работа autovacuum, необходимость VACUUM ANALYZE. Только затем принимается решение о создании нового индекса или перестройке существующего.

После любого изменения план строится повторно. Если время выполнения, количество буферов и тип доступа улучшились, гипотеза подтверждается. Если нет, нужно искать другую причину.

## 75:00–78:00. Связь индексов с планом выполнения

Индексы имеют смысл только в связи с планом выполнения. План показывает, использует ли СУБД индекс, каким способом она это делает и какие дополнительные операции возникают после индексного доступа.

Например, Index Scan может выглядеть хорошо, но если после него выполняется большое количество обращений к heap, запрос всё ещё может быть дорогим. Index Only Scan может быть эффективным, но только при хорошей visibility map. Bitmap Heap Scan может быть нормальным компромиссом для средней селективности.

Поэтому анализ индекса нельзя сводить к вопросу «есть индекс или нет». Нужно смотреть цепочку: условие запроса, структура индекса, выбранный оператор плана, количество строк, буферы, heap fetches и состояние таблицы.

Именно эта связка позволяет понять, почему запрос работает так, как работает.

## 78:00–81:00. Итоги занятия

В финале фиксируется, что индекс — это инструмент ускорения чтения, но не универсальное решение всех проблем производительности. Он эффективен только тогда, когда соответствует запросу, селективности данных и состоянию таблицы.

Составные индексы требуют правильного порядка полей. Покрывающие индексы могут дать Index Only Scan, но для этого нужна актуальная visibility map. После массовых изменений таблицы могут появляться dead tuples, которые ухудшают чтение и требуют обслуживания.

VACUUM, VACUUM FULL, ANALYZE и autovacuum являются не второстепенными административными операциями, а частью нормальной производительности PostgreSQL. Без них оптимизатор может выбирать плохие планы, а индексный доступ может становиться менее эффективным.

Главный практический вывод занятия: диагностика индексов должна опираться на EXPLAIN ANALYZE, BUFFERS, статистику таблиц, состояние dead tuples и проверку фактического плана. Индексы нужно проектировать и оценивать не механически, а через реальные запросы и измеряемые показатели.
