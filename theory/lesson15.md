# Процедурное программирование в SQL

## Превью / аннотация / краткое содержание

Занятие посвящено процедурным расширениям SQL как средству описания алгоритмической логики на стороне базы данных. Центральная проблема связана с ограниченностью чисто декларативного SQL при реализации многошаговых операций, ветвлений, циклов, обработки ошибок и транзакционного поведения. Содержание охватывает процедурные блоки, переменные, условные конструкции, циклы, курсоры и различия процедурных диалектов в разных СУБД.

## Основная часть

### Ограничения декларативного SQL

Декларативный SQL ориентирован на описание результата запроса, а не на пошаговое описание алгоритма его получения. Такая модель эффективна для выборки, фильтрации, соединения и агрегации данных, но становится недостаточной при реализации сложной логики обработки. Многошаговые операции, накопительные расчёты, валидация, обработка исключений и транзакционная координация плохо выражаются в рамках одного SELECT-запроса. Поэтому в SQL-системах используются процедурные расширения, позволяющие дополнять декларативный подход алгоритмическими конструкциями.

#### Многошаговая логика в базе данных

Потребность в процедурном коде возникает тогда, когда операция требует последовательности действий, промежуточных состояний и условного выбора дальнейшего выполнения. Бизнес-правила часто включают проверки, расчёты, ветвления, реакцию на ошибки и управление транзакциями. В чистом SQL такие задачи либо выражаются чрезмерно сложными запросами, либо переносятся в приложение. Процедурные расширения позволяют выполнять часть логики непосредственно на стороне СУБД, где находятся данные и где может быть обеспечена атомарность выполнения.

### Назначение процедурного программирования в SQL

Процедурное программирование в SQL вводит в работу с базой данных элементы императивной логики: переменные, блоки кода, условия, циклы, обработку исключений и управление потоком выполнения. Оно позволяет выполнять вычисления рядом с данными и снижать количество обращений между приложением и сервером базы данных. Такой подход особенно полезен для массовых операций, централизованных бизнес-правил и транзакционных сценариев. Процедурный код расширяет возможности SQL, но требует строгого контроля сложности, производительности и архитектурных границ.

#### Преимущества процедурных расширений

Процедурные расширения обеспечивают выполнение логики на стороне базы данных, что может повышать производительность при массовой обработке. Они позволяют централизовать бизнес-правила и уменьшить дублирование кода в разных приложениях, обращающихся к одной базе. Наличие транзакционного контекста делает процедурный код удобным для операций, где несколько действий должны выполняться атомарно. Дополнительное преимущество связано с прямым доступом к данным, поскольку промежуточные операции не требуют постоянной передачи данных во внешний прикладной слой.

#### Риски процедурного подхода

Процедурный код в базе данных может повышать сложность сопровождения, особенно если логика распределяется между приложением и СУБД без чётких границ ответственности. Инструменты отладки процедурных языков часто ограничены по сравнению с языками общего назначения. Код оказывается привязанным к конкретной СУБД и её диалекту, что затрудняет переносимость между платформами. Неконтролируемое накопление процедур, функций и триггерной логики приводит к эффекту «толстой базы», при котором база данных становится носителем значительной части бизнес-логики.

### Процедурные диалекты в разных СУБД

Разные СУБД используют собственные процедурные расширения SQL, сохраняя общие идеи, но различаясь синтаксисом и деталями реализации. В PostgreSQL используется PL/pgSQL, в Microsoft SQL Server используется T-SQL, в Oracle применяется PL/SQL, а MySQL имеет собственный процедурный синтаксис. Общее ядро этих языков включает блочную структуру кода, переменные, управление потоком выполнения, исключения и транзакционные механизмы. Различия проявляются в форме условных операторов, обработке ошибок, управлении курсорами, типах данных и поддержке отдельных конструкций.

#### Сходства процедурного SQL

Процедурные диалекты разных СУБД сходны по базовым концепциям, поскольку все они решают задачу алгоритмизации работы с данными. В них присутствует блоковая организация кода, разделение объявления переменных и исполняемой части, условные конструкции и циклы. Обработка исключений позволяет выделять нормальный путь выполнения и реакции на ошибки. Транзакционная логика обеспечивает связь процедурного кода с фиксацией, откатом и атомарностью операций. Эти сходства формируют общий понятийный каркас процедурного программирования в SQL.

#### Различия процедурных диалектов

Различия между диалектами затрагивают не только синтаксис, но и поведение отдельных механизмов. Обработка ошибок в PL/pgSQL, T-SQL и PL/SQL организована по-разному, что влияет на структуру надёжного кода. Управление транзакциями также зависит от возможностей конкретной СУБД и допустимости отдельных команд внутри процедурных объектов. Массивы, сложные типы, табличные параметры и курсоры имеют платформенные особенности. Поэтому перенос процедурного кода между СУБД требует не механической замены ключевых слов, а анализа семантики выполнения.

### Структура процедурного блока

Процедурный блок является базовой единицей организации процедурного кода. В общем виде он включает секцию объявления переменных, исполняемую часть и, при наличии поддержки, секцию обработки исключений. Секция DECLARE задаёт имена, типы и начальные значения переменных. Секция BEGIN содержит основную последовательность операторов, включая присваивания, SQL-команды, ветвления и циклы. Секция EXCEPTION описывает реакцию на ошибки и позволяет отделить штатное выполнение от обработки исключительных ситуаций.

#### Анонимный процедурный блок

Анонимный процедурный блок используется для выполнения процедурной логики без создания постоянной функции или процедуры в базе данных. В PostgreSQL такая конструкция обычно реализуется через DO-блок, внутри которого могут объявляться переменные и выполняться операторы PL/pgSQL. Анонимный блок удобен для учебных примеров, административных задач и одноразовых операций обработки данных. Он демонстрирует общую модель процедурного выполнения: объявление состояния, пошаговое изменение переменных, выполнение SQL-операторов и завершение блока.

### Переменные и присваивание значений

Переменные позволяют хранить промежуточные результаты, параметры вычислений и значения, полученные из запросов. В процедурных диалектах SQL переменная объявляется с указанием типа данных и при необходимости начального значения. Присваивание может выполняться через специальные операторы языка или через SELECT, если значение берётся из таблицы. В T-SQL переменные обозначаются префиксом @, а присваивание может выполняться через SET или SELECT. В PL/pgSQL используется собственный синтаксис объявления и присваивания, связанный с блочной структурой языка.

#### Роль переменных в процедурной логике

Переменные обеспечивают переход от декларативного запроса к пошаговой обработке данных. Они позволяют накапливать значения, сравнивать промежуточные результаты, хранить флаги выполнения и передавать состояние между частями блока. Через переменные реализуются вычисления, условия и параметры дальнейших SQL-операций. При работе с запросами переменная может принимать значение отдельного столбца или результата вычисления. Корректное использование переменных требует контроля типов, области видимости и момента присваивания значения.

### Условные конструкции

Условные операторы позволяют выбирать ветвь выполнения в зависимости от результата логического выражения. В процедурных языках SQL используются конструкции IF, ELSE, ELSIF или их платформенные аналоги. Они обеспечивают выполнение одного блока кода при выполнении условия и другого блока при его невыполнении. В PostgreSQL и Oracle используется компактная форма с ELSIF, тогда как в T-SQL ветвления строятся через IF, ELSE и вложенные блоки BEGIN END. Различие синтаксиса не меняет общей функции конструкции: управление направлением выполнения.

#### Ветвление и бизнес-правила

Условные конструкции особенно значимы для реализации бизнес-правил, где дальнейшая операция зависит от значений данных. Через IF можно выполнять проверки допустимости, выбирать способ расчёта, прекращать выполнение или запускать дополнительные действия. В процедурной логике условие связывает данные с алгоритмическим решением. При этом чрезмерное количество вложенных условий снижает читаемость и усложняет сопровождение. Поэтому ветвления должны описывать существенные правила предметной области, а не скрывать неструктурированную логику.

### Циклы в процедурном SQL

Циклы используются для повторного выполнения блока операторов до наступления условия завершения или для обхода набора значений. Конструкция LOOP представляет цикл с явным условием выхода, при котором завершение задаётся оператором EXIT или его аналогом. Цикл FOR применяется для итерации по диапазону значений или по результатам запроса, что делает его удобным для последовательной обработки строк. Наличие циклов расширяет возможности SQL, но не отменяет приоритет множественных операций, поскольку построчная обработка может быть менее эффективной, чем set-based подход.

#### LOOP и явное завершение

LOOP описывает повторяемый блок, который потенциально может выполняться бесконечно, если условие выхода не задано корректно. В PostgreSQL и Oracle завершение такого цикла может выполняться через EXIT или EXIT WHEN. В T-SQL аналогичная задача чаще решается через WHILE и BREAK, поскольку синтаксис LOOP в таком виде не поддерживается. Явное условие выхода является критическим элементом цикла, так как ошибка в нём может привести к зависанию операции или чрезмерной нагрузке на сервер базы данных.

#### FOR и итерация по данным

FOR-цикл применяется для обхода числового диапазона или строк результата запроса. В PostgreSQL и Oracle такая конструкция позволяет последовательно выполнять действия для каждого значения диапазона или каждой записи, полученной SELECT-запросом. Это удобно при процедурной обработке набора строк, когда требуется выполнить индивидуальную проверку или действие. Однако использование FOR для работы с большими объёмами данных должно быть обоснованным. В SQL предпочтение часто отдаётся множественным операциям, если они выражают ту же логику без построчной итерации.

### Операторы прерывания цикла

Операторы прерывания цикла обеспечивают управляемое завершение повторения до естественного окончания итераций. В PostgreSQL и Oracle применяется EXIT, в том числе форма EXIT WHEN с условием. В MS SQL Server используется BREAK, который завершает выполнение текущего цикла. В некоторых диалектах возможно использование именованных меток, позволяющих выходить из конкретного вложенного цикла. Эти механизмы необходимы для предотвращения лишних итераций и реализации сценариев, где дальнейшее повторение теряет смысл после достижения заданного состояния.

#### Управление вложенными циклами

Вложенные циклы требуют дополнительной дисциплины управления, поскольку выход из внутреннего цикла не всегда означает завершение внешнего. Метки цикла позволяют явно указать, какой уровень повторения должен быть прерван. Такая конструкция полезна при сложной обработке наборов данных, но может снижать читаемость кода. Чем больше уровней вложенности и точек выхода, тем выше риск логической ошибки. Поэтому операторы прерывания должны использоваться как средство явного контроля потока, а не как замена структурированному проектированию алгоритма.

### Курсоры как механизм построчной обработки

Курсор представляет собой механизм последовательного обхода результата запроса. Он используется, когда данные необходимо обрабатывать по строкам, а каждая строка требует индивидуальных действий. Общая схема работы с курсором включает объявление курсора для SELECT-запроса, открытие, получение строк через FETCH, обработку полученных значений и закрытие курсора. В SQL Server дополнительно используется освобождение ресурсов через DEALLOCATE. Курсоры дают высокий уровень контроля, но могут быть менее производительными по сравнению с множественными SQL-операциями.

#### Жизненный цикл курсора

Жизненный цикл курсора начинается с объявления, где задаётся запрос, формирующий набор строк. После открытия курсор становится доступным для последовательного чтения. Оператор FETCH извлекает текущую или следующую строку и помещает значения в переменные. После завершения обработки курсор должен быть закрыт, а в отдельных СУБД дополнительно освобождён. Незакрытые курсоры удерживают ресурсы и могут негативно влиять на производительность, поэтому управление курсором является частью корректной процедурной дисциплины.

### Обработка исключений и транзакционная логика

Обработка исключений позволяет процедурному блоку реагировать на ошибки, не сводя выполнение к аварийному завершению без контроля последствий. В PL/pgSQL секция EXCEPTION может содержать обработчики, привязанные к условию ошибки. В T-SQL похожую роль выполняют конструкции TRY CATCH, а в Oracle используется собственная модель обработки исключений. Связь исключений с транзакциями особенно значима, поскольку ошибка может требовать отката, фиксации частичного результата или выполнения компенсирующего действия. Корректная обработка ошибок должна сохранять согласованность данных.

#### Ошибки как часть процедурного сценария

В процедурной логике ошибка рассматривается не только как сбой, но и как управляемое событие, требующее заданной реакции. Например, нарушение ограничения, отсутствие данных или конфликт транзакций могут быть обработаны через отдельный блок. Такой подход позволяет отделить основную бизнес-операцию от аварийных ветвей и сделать поведение базы предсказуемым. При этом обработчик исключений не должен скрывать критические ошибки без фиксации последствий. Его задача состоит в сохранении корректного состояния данных и понятной логике завершения операции.

### Архитектурные границы процедурного кода

Процедурное программирование в SQL требует баланса между эффективностью выполнения на стороне базы и переносимостью логики в прикладной слой. Код базы данных целесообразен там, где операции тесно связаны с данными, транзакциями и массовой обработкой. Логика, зависящая от пользовательских сценариев, интерфейсов или внешних сервисов, чаще должна оставаться в приложении. Нарушение этого разграничения приводит к трудностям сопровождения и росту зависимости от конкретной СУБД. Процедурный SQL должен использоваться как специализированный инструмент, а не как универсальная замена прикладному программированию.

#### Критерии применения процедурного SQL

Процедурный SQL оправдан при необходимости выполнить массовую операцию рядом с данными, обеспечить атомарность нескольких связанных действий или централизовать правило, одинаковое для всех приложений. Он также полезен при административных задачах, миграциях, обработке исключений и проверках, которые должны выполняться внутри транзакции. Его применение менее оправдано при сложной доменной логике, требующей развитого тестирования, модульности и независимости от СУБД. Критерием выбора становится соотношение производительности, сопровождаемости, переносимости и прозрачности архитектуры.

## Итоговые выводы

Процедурное программирование в SQL дополняет декларативный подход средствами алгоритмической обработки данных.

Общими элементами процедурных диалектов являются блоки кода, переменные, ветвления, циклы, исключения и транзакционная логика.

Различия между PL/pgSQL, T-SQL, PL/SQL и другими диалектами затрагивают синтаксис, обработку ошибок, курсоры и управление транзакциями.

Циклы и курсоры обеспечивают построчную обработку, но должны применяться с учётом производительности и предпочтительности множественных SQL-операций.

Главный результат освоения занятия состоит в понимании процедурного SQL как инструмента управляемой логики на стороне базы данных с чёткими архитектурными ограничениями.

### 00:00–03:00. Организационное начало

В начале занятия идёт подключение участников и короткая проверка связи. Тема занятия — процедурное программирование в SQL. Это занятие находится внутри третьего модуля курса, где уже рассматривались индексы, транзакции, представления, последовательности и триггеры. Теперь фокус смещается на процедурные возможности SQL, то есть на те механизмы, которые позволяют писать не только декларативные запросы, но и более алгоритмический код.

### 03:00–06:00. Место занятия в модуле

На карте модуля выделяется тема «Процедурное программирование в SQL». Она находится рядом с темами пользовательских функций, хранимых процедур и работы с представлениями, последовательностями и триггерами. Это означает, что процедурное программирование рассматривается как связующее звено между обычными SQL-запросами и более сложной логикой, которая может выполняться внутри базы данных.

В рамках занятия предполагается разобрать, зачем нужны процедурные расширения, какие у них преимущества и ограничения, чем отличаются процедурные диалекты в разных СУБД, как устроены блоки кода, переменные, условия, циклы и курсоры. После теоретической части выполняется практическая демонстрация в PostgreSQL.

### 06:00–09:00. Ограничения декларативного SQL

SQL по своей природе является декларативным языком. Он хорошо описывает, какие данные необходимо получить, изменить или удалить, но не всегда удобно описывает, как именно должна выполняться пошаговая логика обработки.

Обычный SELECT отлично подходит для выборки, фильтрации, соединения таблиц, группировки и агрегации. Однако многошаговые операции и бизнес-правила часто не помещаются в один запрос. Например, бывает необходимо сначала получить данные, затем проверить условие, затем выполнить разные действия в зависимости от результата, обработать ошибку и зафиксировать или откатить изменения.

Сложности возникают также при валидации, накопительных расчётах, обработке ошибок и транзакционной логике. Всё это выходит за пределы возможностей «чистого» SQL, если использовать только декларативные конструкции.

### 09:00–15:00. Назначение процедурного программирования

Процедурное программирование добавляет к SQL элементы обычного алгоритмического языка: переменные, условия, циклы, блоки кода, обработку исключений и управление потоком выполнения. Благодаря этому можно описывать последовательность действий, а не только конечный результат запроса.

Преимущества процедурного подхода связаны с тем, что код выполняется на стороне базы данных. Это может давать максимум производительности, особенно при массовых операциях, потому что данные не нужно постоянно передавать между приложением и сервером базы данных. Сложная логика становится доступна напрямую внутри СУБД.

Ещё одно преимущество — централизация бизнес-правил. Если несколько приложений работают с одной базой данных, часть правил можно разместить на стороне базы, чтобы они применялись одинаково независимо от клиента. Также процедурный код удобен там, где требуется атомарность: несколько действий должны выполняться внутри одной транзакции.

Но у процедурного программирования есть и ограничения. Процедуры могут становиться сложными, инструменты отладки обычно менее удобны, чем в языках общего назначения. Код привязывается к конкретной СУБД, потому что процедурные диалекты отличаются. При неправильном использовании появляется риск «толстой базы», когда слишком большая часть бизнес-логики переносится внутрь СУБД. Кроме того, неудачно написанный процедурный код может ухудшать производительность.

Главная идея: процедурные расширения дают большие возможности, но требуют аккуратного использования.

### 15:00–21:00. Сходства процедурного SQL в разных СУБД

Несмотря на различия между СУБД, процедурные расширения имеют общие черты. Во всех основных системах есть блочная структура кода. Обычно код делится на область объявления переменных и исполняемую часть.

Также в разных СУБД поддерживаются переменные и управление потоком выполнения: условия, ветвления, циклы. Есть механизмы обработки исключений, то есть реакции на ошибки. Кроме того, процедурный SQL тесно связан с транзакциями, потому что часто используется для выполнения набора связанных операций с данными.

Эти общие элементы присутствуют в PL/pgSQL для PostgreSQL, T-SQL для Microsoft SQL Server, PL/SQL для Oracle и процедурном синтаксисе MySQL. Концептуально они решают одну задачу: позволяют описывать алгоритм работы с данными внутри базы.

### 21:00–24:00. Различия процедурного SQL в разных СУБД

При этом между диалектами есть важные различия. Отличаются механизмы обработки ошибок, синтаксис языка, управление транзакциями, поддержка массивов и сложных типов, а также работа с табличными параметрами.

Например, в PostgreSQL используется PL/pgSQL, в SQL Server — T-SQL, в Oracle — PL/SQL. У каждого языка свои правила объявления переменных, свои формы условных конструкций, свои способы обработки исключений и свои особенности курсоров.

Поэтому процедурный код нельзя просто механически перенести из одной СУБД в другую. Нужно учитывать не только синтаксис, но и поведение конкретной системы: как она выполняет блоки, как управляет транзакциями, как обрабатывает ошибки и как работает с типами данных.

### 24:00–27:00. PostgreSQL: PL/pgSQL

В PostgreSQL процедурное программирование чаще всего реализуется через PL/pgSQL. Это язык, встроенный в PostgreSQL и предназначенный для написания функций, процедур, триггерных функций и анонимных блоков.

PL/pgSQL позволяет объявлять переменные, выполнять SQL-операторы, использовать условия, циклы, исключения и работать с результатами запросов. В отличие от обычного SQL-запроса, здесь можно описывать последовательность действий.

Для демонстрации дальше используется PostgreSQL и среда pgAdmin. Большая часть практики строится на анонимных блоках DO, потому что они позволяют выполнять процедурный код без создания постоянной функции в базе данных.

### 27:00–30:00. Пример T-SQL в SQL Server

В SQL Server процедурный язык называется T-SQL. На слайде приводится пример объявления переменных и условного вывода результата:

declare @a int = 7,

@b int = 5,

@result int;

set @result = @a + @b;

begin

if @result > 10

print concat('Result is greater than 10: ', @result);

else

print concat('Result is ', @result);

end;

В этом примере объявляются три переменные: два исходных числа и переменная для результата. Затем результат вычисляется через сложение, после чего проверяется условие. Если значение больше 10, выводится одно сообщение, иначе другое.

Этот пример показывает базовую идею процедурного SQL: можно хранить промежуточные значения, выполнять присваивание и выбирать ветку выполнения в зависимости от условия.

### 30:00–33:00. Универсальная структура процедурного блока

Процедурный блок обычно имеет несколько частей. Сначала идёт секция объявления переменных:

DECLARE

variable1 TYPE := initial_value;

variable2 TYPE;

Затем начинается исполняемый блок:

BEGIN

-- основной код

variable2 := variable1 + 10;

IF variable2 > 20 THEN

-- выполнить действие

ELSE

-- другое действие

END IF;

SELECT ... INTO variable3;

UPDATE table SET ...;

В некоторых СУБД после основной части может быть секция обработки исключений:

EXCEPTION

WHEN error_condition THEN

-- обработка ошибок

END;

Такой блок показывает общую структуру процедурного программирования: объявление состояния, выполнение действий, условная логика, SQL-операторы и возможная обработка ошибок.

### 33:00–36:00. Переменные в T-SQL

В SQL Server переменные обозначаются через символ @. Например:

declare @x int = 10;

declare @y decimal(10,2);

Переменные могут получать значения через SET:

set @x = @x + 1;

Также значение можно присвоить из результата запроса:

select @y = amount

from payments

where id = 1;

Допускается объявление переменной со значением по умолчанию:

declare @x int = 5;

Переменные нужны для хранения промежуточных результатов, значений из запросов, счётчиков, флагов и параметров дальнейшего выполнения.

### 36:00–39:00. Условные конструкции в разных диалектах

Условные конструкции отличаются по синтаксису. В PostgreSQL, MySQL и Oracle может использоваться форма:

if условие_1 then

-- код 1

elsif условие_2 then

-- код 2

elsif условие_3 then

-- код 3

else

-- код иначе

end if;

В SQL Server конструкция выглядит иначе:

if условие_1

begin

-- код 1

end

else if условие_2

begin

-- код 2

end

else if условие_3

begin

-- код 3

end

else

begin

-- код иначе

end

Смысл одинаковый: выполняется проверка условий, и в зависимости от результата выбирается одна ветка кода. Отличие заключается в синтаксисе и способе оформления блоков.

### 39:00–42:00. LOOP как цикл с явным условием выхода

В PostgreSQL и некоторых других диалектах есть конструкция LOOP. Это цикл, который сам по себе может быть бесконечным, поэтому внутри него нужно явно задать условие выхода:

LOOP

-- инструкции для выполнения

IF <условие выхода> THEN

EXIT;

END IF;

END LOOP;

В SQL Server такой LOOP не поддерживается. Там чаще используется WHILE.

LOOP удобен, когда заранее неизвестно количество повторений, но известно условие, при котором цикл должен завершиться. При этом важно не забыть условие выхода, иначе можно получить бесконечный цикл.

### 42:00–45:00. FOR как цикл по диапазону или результатам запроса

Цикл FOR используется для итерации по диапазону значений или по результатам запроса. Для диапазона в PostgreSQL и Oracle структура может выглядеть так:

FOR counter_variable IN <start_value>..<end_value> LOOP

-- инструкции для выполнения

END LOOP;

Также FOR может проходить по результатам запроса:

FOR record_variable IN QUERY LOOP

-- действия с каждой строкой результата

END LOOP;

Это удобно, когда нужно обработать набор строк по одной. Но необходимо помнить, что в SQL обычно предпочтительнее множественная обработка данных. Если задачу можно решить одним UPDATE, INSERT INTO SELECT или другим set-based запросом, это часто эффективнее, чем построчная итерация.

### 45:00–48:00. Операторы прерывания цикла

Для выхода из цикла используются разные операторы. В PostgreSQL:

EXIT;

EXIT WHEN <condition>;

В SQL Server используется:

BREAK;

В Oracle также есть:

EXIT;

EXIT WHEN <condition>;

Для выхода из именованного блока или цикла может использоваться конструкция вида:

LEAVE label_name;

Такие операторы нужны, чтобы завершить цикл до его естественного окончания. Например, если найдено нужное значение или достигнуто условие остановки, дальнейшие итерации больше не имеют смысла.

### 48:00–51:00. Курсоры

Курсор — это механизм построчной обработки результата запроса. Он используется, когда результат SELECT нужно обходить строка за строкой.

В PostgreSQL и MySQL общая схема такая:

declare cursor_name cursor for

select ...;

open cursor_name;

fetch cursor_name into var1, var2;

close cursor_name;

В SQL Server синтаксис немного отличается:

declare cursor_name cursor for

select ...;

open cursor_name;

fetch next from cursor_name into @v1, @v2;

close cursor_name;

deallocate cursor_name;

Курсор сначала объявляется, затем открывается, после этого из него извлекаются строки через FETCH. После завершения работы курсор закрывается. В SQL Server дополнительно освобождаются ресурсы через DEALLOCATE.

Курсоры дают контроль над построчной обработкой, но их нужно использовать осторожно. По производительности они часто хуже множественных SQL-операций.

### 51:00–54:00. Практика: подготовка таблицы

В практической части открывается pgAdmin и создаётся учебная таблица:

drop table if exists demo_numbers;

create table demo_numbers(

id int,

val text

);

Затем в неё вставляются тестовые строки:

insert into demo_numbers values

(1, 'one'),

(2, 'two'),

(3, 'three');

Эта таблица используется дальше для демонстрации блоков, переменных, условий, циклов, обхода строк и курсоров.

### 54:00–57:00. Практика: блоки и переменные

Далее демонстрируется анонимный блок DO в PostgreSQL. Внутри блока объявляются переменные:

do $$

declare

a int := 10;

b int := 20;

c int;

begin

c := a + b;

raise notice 'result = %', c;

end;

$$;

Смысл примера: в секции DECLARE задаются переменные, в секции BEGIN выполняется присваивание и вывод сообщения через RAISE NOTICE.

После выполнения блока результат выводится в нижней панели сообщений. Это показывает, что процедурный код может выполнять вычисления и выводить диагностическую информацию без создания отдельной таблицы.

Также демонстрируются примеры присваивания значения из запроса, когда результат SELECT помещается в переменную.

### 57:00–60:00. Практика: CASE и условная логика

Дальше показывается конструкция CASE как оператор. Например:

do $$

declare

x int := 2;

begin

case

when x = 1 then

raise notice 'CASE оператор: x = 1';

when x = 2 then

raise notice 'CASE оператор: x = 2';

else

raise notice 'CASE оператор: другое значение';

end case;

end;

$$;

Здесь значение переменной проверяется по нескольким вариантам. Если x = 1, выводится первое сообщение; если x = 2, второе; иначе выполняется ветка else.

Также CASE может использоваться как выражение, то есть не только выбирать блок кода, но и возвращать значение, которое затем присваивается переменной.

### 60:00–63:00. Практика: LOOP, CONTINUE и EXIT

Демонстрируется цикл LOOP. Внутри цикла используется счётчик i, который увеличивается на каждой итерации:

do $$

declare

i int := 0;

begin

raise notice 'Начинаем цикл LOOP...';

loop

i := i + 1;

if i % 2 = 0 then

raise notice 'LOOP: i=% - iteration skipped (continue)', i;

continue;

end if;

raise notice 'LOOP: i=% - выполнена итерация', i;

if i = 7 then

raise notice 'LOOP: EXIT при i=%', i;

exit;

end if;

end loop;

end;

$$;

В этом примере CONTINUE пропускает дальнейшее выполнение текущей итерации, если число чётное. EXIT завершает цикл, когда счётчик достигает значения 7.

В чате появляется вопрос: если EXIT завершает цикл, зачем тогда нужен END LOOP. Ответ по смыслу такой: EXIT управляет выполнением цикла во время работы программы, а END LOOP является синтаксическим завершением блока цикла. Даже если внутри есть выход, сама конструкция цикла должна быть закрыта.

### 63:00–66:00. Практика: WHILE и FOR по диапазону

После LOOP показывается цикл WHILE. Он выполняется до тех пор, пока условие истинно. Пример строится на счётчике и выводе сообщений:

do $$

declare

i int := 1;

begin

raise notice 'Начинаем цикл WHILE...';

while i <= 5 loop

raise notice 'WHILE: i=%', i;

i := i + 1;

end loop;

end;

$$;

Далее демонстрируется FOR по диапазону:

do $$

begin

raise notice 'Начинаем цикл FOR по диапазону...';

for i in 1..5 loop

raise notice 'FOR диапазон: i=%', i;

end loop;

end;

$$;

Этот пример показывает более компактный способ пройти по фиксированному диапазону значений.

### 66:00–69:00. Практика: FOR по результатам запроса и FOREACH

Затем используется FOR по результатам запроса к таблице demo_numbers:

do $$

declare

r record;

begin

raise notice 'Начинаем цикл FOR по данным таблицы demo_numbers...';

for r in

select id, val

from demo_numbers

order by id

loop

raise notice 'FOR таблица: id=%, val=%', r.id, r.val;

end loop;

end;

$$;

Переменная r имеет тип record и на каждой итерации содержит очередную строку результата запроса. Через r.id и r.val можно обращаться к значениям столбцов.

Также начинается демонстрация FOREACH по массиву. Такая конструкция позволяет пройти по элементам массива, например по числам 10, 20, 30.

### 69:00–72:00. Практика: курсор

Далее демонстрируется явная работа с курсором:

do $$

declare

cursor_demo cursor for

select id, val

from demo_numbers

order by id;

rec record;

begin

raise notice 'Начинаю работу с курсором...';

open cursor_demo;

loop

fetch cursor_demo into rec;

exit when not found;

raise notice 'Курсор: id=%, val=%', rec.id, rec.val;

end loop;

close cursor_demo;

raise notice 'Работа курсора завершена.';

end;

$$;

Сначала курсор объявляется для запроса к таблице demo_numbers. Затем он открывается. В цикле выполняется FETCH, который помещает очередную строку в переменную rec. Если строк больше нет, срабатывает exit when not found. После завершения обработки курсор закрывается.

Этот пример показывает полный жизненный цикл курсора: объявление, открытие, чтение, завершение цикла и закрытие.

### 72:00–75:00. Обсуждение производительности курсоров

В чате появляется вопрос о том, правда ли, что производительность курсора ниже, чем у обычных SQL-операций. Ответ по смыслу: да, курсоры часто работают медленнее, потому что обрабатывают строки по одной. SQL изначально рассчитан на множественные операции над наборами данных, и если можно решить задачу одним запросом, это обычно предпочтительнее.

Курсоры оправданы тогда, когда для каждой строки действительно нужна отдельная логика, которую сложно или невозможно выразить множественной операцией. Но использовать курсор просто как привычный цикл по строкам не стоит, особенно на больших объёмах данных.

### 75:00–78:00. Вопросы о хранении SQL-кода

Обсуждается вопрос, как хранить SQL-код и процедурный код. В чате спрашивают, как в компаниях доносят код из Git в базу данных и как организован процесс.

По смыслу объясняется, что код базы данных обычно хранится в репозитории: миграции, SQL-скрипты, функции, процедуры, изменения схемы. Затем эти изменения применяются к базе через процесс деплоя. В компаниях могут использоваться миграционные инструменты, CI/CD, отдельные скрипты или специальные средства для управления изменениями БД.

Также упоминаются инструменты, которые могут помогать в работе с базами данных и схемами: DataGrip, dbdiagram, codekeeper и другие средства для проектирования, просмотра и сопровождения структуры.

### 78:00–81:00. Завершение основной части и вопросы

После практической демонстрации открывается слайд с блоком вопросов. Участникам предлагается задать вопросы по теме занятия. Основные вопросы касаются курсоров, циклов, хранения кода базы данных и практического использования процедурных расширений.

Фиксируется общая мысль: процедурный SQL полезен, но его нужно применять там, где он действительно оправдан. Для массовых операций предпочтительнее обычные SQL-запросы над наборами данных. Для логики, тесно связанной с данными и транзакциями, процедурный код может быть эффективным решением.

### 81:00–88:40. Завершение занятия

В финале участникам предлагается заполнить опрос о занятии. Занятие завершается напоминанием о том, что сообщения читаются и обратная связь учитывается.

Итог занятия сводится к тому, что процедурное программирование расширяет возможности SQL. Оно позволяет использовать переменные, условия, циклы, курсоры и обработку ошибок на стороне базы данных. При этом важно понимать различия между диалектами СУБД и не переносить в базу чрезмерный объём бизнес-логики без необходимости.
