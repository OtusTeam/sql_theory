# Секционирование таблиц в реляционных СУБД

## Превью / аннотация / краткое содержание

Занятие посвящено секционированию таблиц как способу физической и логической организации больших объёмов данных в реляционных СУБД.
 Ключевая проблема связана с повышением производительности запросов, уменьшением объёма читаемых данных и упрощением обслуживания крупных таблиц.
 Материал раскрывает назначение секционирования, стратегии RANGE, LIST, HASH, особенности реализации в PostgreSQL, MS SQL Server, Oracle и MySQL, а также ограничения и условия эффективного применения.
 Отдельное внимание уделяется partition pruning, управлению секциями и преобразованию существующей таблицы в секционированную структуру.

## Основная часть

## Секционирование как механизм организации больших таблиц

Секционирование представляет собой разбиение одной логической таблицы на несколько физических частей, каждая из которых хранит определённый фрагмент данных. Для пользователя такая структура может оставаться единой таблицей, но на уровне хранения и выполнения запросов данные распределяются по секциям. Основная цель механизма состоит в сокращении объёма данных, которые должна читать СУБД при выполнении запроса. При корректном выборе ключа секционирования оптимизатор может обращаться только к релевантным секциям, не просматривая всю таблицу целиком. Поэтому секционирование является не заменой индексам, а самостоятельным способом управления большими массивами данных.

### Назначение секционирования

Секционирование используется для повышения производительности запросов, уменьшения объёма читаемых данных и упрощения обслуживания крупных таблиц. При работе с большими таблицами операции VACUUM FULL, REINDEX, резервное копирование и архивирование могут становиться затратными, поскольку затрагивают слишком большой объём данных. Разделение таблицы на секции позволяет обслуживать не всю структуру сразу, а отдельные части, например данные за конкретный период. Дополнительный эффект связан с переносом архивных секций на более дешёвые носители и упрощением процедур удаления устаревших данных.

## Логическая и физическая модель секционированной таблицы

Секционированная таблица сохраняет единый логический интерфейс, но её данные фактически распределяются между дочерними объектами. Каждая секция содержит строки, удовлетворяющие определённому условию секционирования. Это условие может быть основано на диапазоне значений, списке допустимых значений или хэш-функции. При вставке строки СУБД определяет, в какую секцию она должна попасть, исходя из значения ключа секционирования. Для корректной работы все секции должны иметь одинаковую структуру столбцов, поскольку они представляют части одной логической таблицы.

### Ключ секционирования

Ключ секционирования определяет правило распределения строк между секциями. От его выбора зависит эффективность всей схемы, поскольку именно по этому признаку оптимизатор сможет исключать нерелевантные части таблицы. Наиболее удачным ключом становится поле, которое часто используется в условиях фильтрации и естественно разделяет данные: дата события, идентификатор клиента, регион, категория или иной стабильный атрибут. Если запросы редко фильтруют данные по ключу секционирования, механизм не даст существенного выигрыша. Поэтому секционирование проектируется от характера запросов, а не только от размера таблицы.

## Секционирование через наследование в PostgreSQL

В PostgreSQL до версии 10 секционирование строилось на механизме наследования таблиц. Родительская таблица задавала общую структуру, а дочерние таблицы содержали отдельные части данных. Такой подход позволял реализовать разбиение, но требовал значительного ручного управления: маршрутизации вставок, триггеров, ограничений целостности и планирования секций. Чтение сначала охватывало несколько таблиц, позднее появилась возможность избирательного чтения при определённых условиях. Несмотря на работоспособность, этот механизм считается устаревшим и сложным для сопровождения вне специальных случаев.

### Ограничения наследуемого подхода

Секционирование через наследование плохо подходит для типовых задач, поскольку значительная часть логики переносится на разработчика или администратора базы данных. Вставка строк не распределяется автоматически без дополнительного триггера маршрутизации, а поддержание ограничений требует отдельного контроля. При росте числа секций усложняются обслуживание, проверка корректности и изменение структуры. Такой подход может использоваться в специфических сценариях, но для большинства современных задач предпочтительнее декларативное секционирование. Оно делает структуру секций частью явного синтаксиса СУБД, а не набором вспомогательных соглашений.

## Декларативное секционирование

Декларативное секционирование задаёт разбиение таблицы средствами специального синтаксиса СУБД. При создании таблицы указывается стратегия секционирования и столбец, по которому данные будут распределяться. Далее создаются отдельные секции с условиями попадания строк. Такой подход делает модель секционирования явной: СУБД знает, что таблица состоит из секций, и может использовать эту информацию при планировании запросов. В PostgreSQL это выражается через PARTITION BY RANGE, LIST или HASH, а секции создаются как PARTITION OF основной таблицы.

### Управление секциями

Секции могут создаваться, удаляться, присоединяться и отсоединяться от основной таблицы. В PostgreSQL для присоединения используется ATTACH PARTITION, а для отключения секции от секционированной структуры применяется обратная операция. Это особенно полезно для архивирования: старая секция может быть отсоединена, перенесена или обработана отдельно без полного изменения всей таблицы. Для строк, которые не подходят ни под одно заданное условие, может создаваться секция по умолчанию. Такая секция выполняет роль страховочного контейнера, но её наличие не отменяет необходимости корректно планировать диапазоны или списки значений.

## Реализация секционирования в разных СУБД

Реализация секционирования различается в зависимости от СУБД. PostgreSQL использует декларативную модель PARTITION BY, где секции создаются как дочерние объекты основной таблицы. MS SQL Server строит секционирование через partition function и partition scheme, связывая диапазоны значений с файловыми группами и индексной организацией. Oracle поддерживает развитые операции управления секциями, включая перемещение секции в другой tablespace и обмен секции с обычной таблицей. MySQL поддерживает несколько стратегий, но имеет ограничения по глубине секционирования и подчинённым секциям.

### Сравнение PostgreSQL, MS SQL Server, Oracle и MySQL

PostgreSQL делает акцент на декларативной структуре таблицы и явном создании секций. MS SQL Server тесно связывает секционирование с физической организацией хранения через partition scheme и clustered primary key. Oracle предоставляет расширенные административные операции, такие как MOVE PARTITION и EXCHANGE PARTITION, что делает его удобным для сложных сценариев эксплуатации больших хранилищ. MySQL допускает двухуровневую структуру, где первый уровень может быть RANGE или LIST, а второй уровень связан с HASH. Синтаксис и возможности различаются, поэтому перенос схемы секционирования между СУБД требует адаптации, а не механического копирования.

## Стратегии секционирования

Выбор стратегии секционирования определяет способ распределения строк по секциям. RANGE делит данные по непересекающимся интервалам значений, LIST распределяет строки по заранее заданным спискам, HASH использует хэш-функцию для равномерного распределения, INTERVAL автоматизирует создание новых секций по мере появления новых диапазонов, а KEY является вариантом хэширования по ключу. Не все стратегии поддерживаются одинаково во всех СУБД. RANGE встречается в PostgreSQL, Oracle, MySQL и MS SQL Server; LIST поддерживается в PostgreSQL, Oracle и MySQL; HASH доступен в PostgreSQL, Oracle и MySQL; INTERVAL характерен для Oracle; KEY используется в MySQL.

### RANGE-секционирование

RANGE-секционирование основано на делении строк по непересекающимся диапазонам значений. Типовые ключи для такой стратегии: дата, числовой идентификатор, номер, стоимость или иной упорядоченный атрибут. Каждое значение должно попадать ровно в одну секцию, что требует внимательного задания границ. В PostgreSQL для секций используется конструкция FOR VALUES FROM ... TO .... Стратегия особенно удобна для временных данных, например журналов, заказов, событий и операций, где естественным образом выделяются периоды хранения и анализа.

## LIST- и HASH-секционирование

LIST-секционирование распределяет данные по фиксированным наборам значений. Оно подходит для категориальных признаков: регионов, типов клиентов, статусов, подразделений или иных дискретных групп. Такая стратегия удобна, когда значения заранее известны и логически образуют устойчивые группы. HASH-секционирование решает другую задачу: оно распределяет строки по секциям на основе результата хэш-функции. При этом пользователь обычно не задаёт смысловые группы, а получает более равномерное распределение данных между секциями.

### MODULUS и REMAINDER в HASH-секционировании

В PostgreSQL секционирование по хэшу задаётся через параметры MODULUS и REMAINDER. MODULUS определяет общее количество секций, а REMAINDER задаёт номер остатка, соответствующий конкретной секции. Значения ключа проходят через хэш-функцию, после чего результат распределяется по остаткам от деления. Такой способ полезен, когда данные нужно разложить равномерно, но нет естественного диапазона или списка значений. Ограничение HASH-подхода состоит в том, что он хуже подходит для смыслового архивирования и выборочного удаления по временным периодам.

## Partition pruning

Partition pruning, или исключение секций, представляет собой механизм, при котором оптимизатор исключает из плана запроса секции, не содержащие нужных данных. Это ключевое условие практической эффективности секционирования. Если запрос содержит фильтр по ключу секционирования, СУБД может определить, какие секции не имеют отношения к результату, и не обращаться к ним. В PostgreSQL, Oracle и MySQL исключение секций обычно работает на этапе планирования запроса. В MS SQL Server возможны механизмы как на этапе планирования, так и во время выполнения через runtime elimination.

### Условия эффективности partition pruning

Partition pruning работает только при наличии фильтрации по ключу секционирования или по выражению, которое оптимизатор способен соотнести с границами секций. Если фильтр вычисляется слишком поздно или ключ секционирования скрыт внутри функции, исключение секций может не сработать. Например, условие вида date(created_at) = '2024-01-01' может быть менее эффективно, чем прямое сравнение диапазона по created_at. В PostgreSQL требуется активный параметр enable_partition_pruning. Практический вывод состоит в том, что секционирование должно сопровождаться корректным стилем написания запросов.

## Ограничения секционирования

Секционирование имеет ряд системных ограничений, которые необходимо учитывать при проектировании. Все секции должны иметь тот же набор столбцов, что и основная секционированная таблица. Ключ секционирования должен входить в состав первичного ключа или уникального ограничения, иначе СУБД не сможет гарантировать уникальность в пределах всей логической таблицы. Уникальность поддерживается только с учётом ключа секционирования. Нельзя создать уникальный индекс по другим полям, если они не включают столбцы секционирования.

### Ограничения проектирования и миграции

В PostgreSQL нельзя декларативно секционировать уже существующую таблицу одной простой командой. Для такой задачи требуется миграционный сценарий: подготовить ограничения, создать новую секционированную таблицу с той же структурой, присоединить старую таблицу как секцию или перенести данные. При этом необходимо валидировать ограничения и контролировать соответствие диапазонов. Секционирование также не должно применяться механически к любой большой таблице. Если таблица велика, но запросы хорошо покрываются индексами и не требуют чтения крупных диапазонов, секционирование может не дать ожидаемого эффекта.

## Преобразование существующей таблицы в секционированную

Преобразование большой существующей таблицы требует аккуратной последовательности действий. Сначала добавляется CHECK-ограничение, соответствующее будущему диапазону секции; оно может быть создано как NOT VALID, чтобы не блокировать процесс полной проверкой на первом шаге. Затем создаётся новая таблица с той же структурой, но уже с PARTITION BY. После этого текущая таблица может быть присоединена как секция через ATTACH PARTITION. На завершающем этапе выполняется валидация ограничения, подтверждающая соответствие данных заявленному диапазону.

### Практическая логика миграции

Миграция к секционированной модели должна минимизировать блокировки и риск нарушения целостности данных. Поэтому важно заранее определить диапазоны, проверить данные на соответствие будущим секциям и подготовить ограничения. Если данные уже распределены по периодам или категориям, их проще переносить партиями. При наличии активной записи в таблицу необходимо учитывать конкурентный доступ и порядок переключения приложения на новую структуру. Смысл миграции состоит не просто в создании секций, а в сохранении корректности данных и предсказуемости работы системы после изменения физической организации таблицы.

## Итоговые выводы

Секционирование является механизмом разбиения большой логической таблицы на физические секции для повышения эффективности запросов и обслуживания данных.

Эффективность секционирования определяется выбором ключа секционирования и способностью запросов использовать partition pruning.

Основные стратегии секционирования различаются по способу распределения строк: RANGE по диапазонам, LIST по спискам значений, HASH по результату хэширования.

Реализация секционирования зависит от конкретной СУБД, поэтому синтаксис и эксплуатационные возможности PostgreSQL, MS SQL Server, Oracle и MySQL необходимо рассматривать отдельно.

Главный результат освоения занятия состоит в понимании секционирования как проектного решения, требующего связи между объёмом данных, характером запросов, ограничениями целостности и процедурой сопровождения таблиц.

В центре темы находится вопрос: как работать с таблицами, которые становятся слишком большими для удобного чтения, обслуживания, архивирования и резервного копирования.

## 09:00–12:00. Постановка темы: секционирование таблиц

Секционирование — это разбиение одной большой логической таблицы на несколько физических частей. Для пользователя и SQL-запросов таблица может выглядеть как единый объект, но на уровне хранения данные распределяются по отдельным секциям. Каждая секция содержит только часть строк, определяемую правилом секционирования.

Главная идея секционирования состоит в том, что СУБД не всегда должна читать всю таблицу. Если данные распределены по секциям и запрос содержит условие по ключу секционирования, оптимизатор может исключить нерелевантные секции из чтения. За счёт этого уменьшается объём обрабатываемых данных и повышается предсказуемость работы с большими таблицами.

Секционирование не заменяет индексы. Индекс помогает быстро найти строки внутри структуры хранения, а секционирование меняет саму физическую организацию таблицы. На практике эти механизмы часто используются вместе.

## 12:00–15:00. Зачем нужно секционирование

Секционирование нужно прежде всего для повышения производительности запросов. Если таблица содержит очень большой объём строк, чтение всей таблицы или большого диапазона становится дорогой операцией. Разбиение на секции позволяет ограничить чтение только той частью данных, которая действительно нужна запросу.

Вторая причина — уменьшение объёма читаемых данных. Например, если данные разбиты по месяцам, а запрос обращается только к январю, СУБД может не читать секции за остальные месяцы. Это особенно полезно для журналов, событий, заказов, финансовых операций и других временных данных.

Третья причина — упрощение обслуживания больших таблиц. Операции VACUUM FULL, REINDEX, pg_dump --jobs=n и другие административные задачи можно выполнять по частям. Также секционирование упрощает архивирование данных, перенос старых секций на более дешёвые носители и организацию резервного копирования.

## 15:00–18:00. Логика секционирования на примере MS SQL Server

На примере MS SQL Server показывается общая идея: исходная таблица содержит множество строк, а затем эти строки распределяются по нескольким физическим частям. Визуально одна таблица как будто раскладывается на несколько отдельных таблиц, каждая из которых содержит свою часть строк.

Для пользователя это не обязательно означает работу с несколькими самостоятельными таблицами. В нормальной модели секционирования сохраняется единая логическая таблица. Запросы продолжают обращаться к одному имени таблицы, но СУБД внутри понимает, где физически лежат нужные строки.

Практический смысл такого подхода в том, что при грамотном ключе секционирования база может быстрее обрабатывать запросы, а администратор может проще управлять большими объёмами данных. Особенно это заметно там, где данные естественно делятся на периоды, диапазоны идентификаторов или категории.

## 18:00–21:00. Секционирование через наследование в PostgreSQL до версии 10

До появления декларативного секционирования в PostgreSQL использовался подход через наследование таблиц. Родительская таблица задавала общую структуру, а дочерние таблицы выступали как отдельные части этой структуры. Каждая дочерняя таблица хранила определённый фрагмент данных.

Такой механизм позволял реализовать секционирование, но требовал большого количества ручной настройки. Чтение сначала охватывало все таблицы, позднее появилась возможность избирательного чтения, но она зависела от условий и настроек. Автоматическая маршрутизация вставок не выполнялась сама по себе, поэтому для распределения строк по дочерним таблицам требовались триггеры.

Главный недостаток этого подхода — сложность сопровождения. Нужно самостоятельно контролировать ограничения, маршрутизацию, создание новых секций и корректность распределения данных. Поэтому секционирование через наследование считается устаревшим механизмом и рекомендуется только для специфических случаев.

## 21:00–24:00. Декларативное секционирование

Декларативное секционирование — современный способ описать секционированную таблицу средствами самой СУБД. Разработчик явно указывает, что таблица должна быть разбита на секции, и задаёт стратегию распределения строк: по диапазону, по списку значений или по хэшу.

В PostgreSQL такая логика задаётся при создании таблицы через PARTITION BY. Общая форма выглядит так:

CREATE TABLE имя_таблицы (

список_столбцов

)

PARTITION BY { RANGE | LIST | HASH } (секционный_столбец);

После создания основной таблицы создаются отдельные секции:

CREATE TABLE имя_секции

PARTITION OF имя_таблицы

FOR VALUES (условие_секционирования);

Такой синтаксис делает секционирование явным. СУБД понимает, что перед ней не набор произвольных таблиц, а единая секционированная структура.

## 24:00–27:00. Секции по умолчанию и удаление секций

Для секционированной таблицы можно создать секцию по умолчанию. Она принимает строки, которые не попали ни в одну из явно заданных секций. Это полезно как страховочный механизм, чтобы данные не терялись при вставке значений, для которых заранее не была создана подходящая секция.

Пример:

CREATE TABLE имя_секции_по_умолчанию

PARTITION OF имя_таблицы DEFAULT;

Удаление секции выполняется как удаление соответствующей таблицы:

DROP TABLE имя_секции;

Такой подход удобен для архивирования. Например, если данные разбиты по месяцам, устаревшую секцию можно отсоединить или удалить значительно проще, чем удалять миллионы строк из одной большой таблицы. Это одно из важных эксплуатационных преимуществ секционирования.

## 27:00–30:00. Присоединение и отсоединение секций

Секции можно не только создавать сразу вместе с таблицей, но и присоединять к уже существующей секционированной структуре. В PostgreSQL для этого используется ATTACH PARTITION.

Общая форма:

ALTER TABLE имя_секционированной_таблицы

ATTACH PARTITION имя_таблицы_секции

FOR VALUES <условие_секционирования>;

Смысл операции в том, что обычная таблица или заранее подготовленная таблица становится секцией основной секционированной таблицы. Это важно при миграциях, архивировании и работе с историческими данными.

Отсоединение секции позволяет вывести часть данных из общей структуры, не удаляя их физически. Например, старую секцию можно отделить, сохранить как архивную таблицу, перенести в другое хранилище или обработать отдельно. Поэтому секционирование полезно не только для чтения, но и для жизненного цикла данных.

## 30:00–33:00. Вопрос о пользе секционирования в MySQL

В чате появляется вопрос о том, какой профит даёт секционирование в MySQL, если внешне всё равно остаётся одна таблица и разработчик может не сталкиваться с партициями напрямую. Ответ строится вокруг различия между логическим и физическим уровнем.

С точки зрения приложения действительно может оставаться одна таблица. Запросы продолжают выполняться к одному имени таблицы. Но внутри СУБД данные физически распределены по частям, и оптимизатор может работать не со всей таблицей, а только с нужными секциями. Поэтому польза проявляется не в том, что разработчик вручную ходит по разным таблицам, а в том, что СУБД сокращает объём работы.

Также секционирование полезно для многопоточности и обслуживания. Когда таблица разбита на части, отдельные операции могут быть проще, быстрее или безопаснее для выполнения. Это особенно актуально для больших таблиц, где обычный индекс уже не решает все проблемы.

## 33:00–36:00. Декларативное секционирование в MS SQL Server

В MS SQL Server секционирование устроено иначе, чем в PostgreSQL. Там используются partition function и partition scheme. Partition function определяет границы распределения данных, а partition scheme связывает эти границы с физическим размещением.

Создание секционированной таблицы связано с первичным ключом и схемой секционирования:

CREATE TABLE имя_таблицы (

список_столбцов,

CONSTRAINT имя_PK PRIMARY KEY CLUSTERED (секционный_столбец)

ON имя_схемы(секционный_столбец)

)

ON имя_схемы (секционный_столбец);

Добавление новой партиции в MS SQL Server может выполняться через изменение partition function:

ALTER PARTITION FUNCTION имя_функции()

SPLIT RANGE (новая_граница);

Здесь видно важное отличие: в SQL Server секционирование тесно связано с физической схемой хранения и индексной организацией.

## 36:00–39:00. Декларативное секционирование в Oracle

В Oracle секционирование поддерживает развитые административные операции. Один из примеров — перемещение секции в другой tablespace:

ALTER TABLE имя_таблицы

MOVE PARTITION имя_секции

TABLESPACE имя_tablespace;

Это позволяет размещать разные части данных на разных физических носителях или в разных областях хранения. Например, актуальные данные можно держать на более быстрых дисках, а старые архивные секции переносить на более дешёвое хранилище.

Ещё одна операция Oracle — обмен секции с обычной таблицей:

ALTER TABLE имя_таблицы

EXCHANGE PARTITION имя_секции

WITH TABLE имя_обычной_таблицы

WITHOUT VALIDATION;

Такой механизм используется для быстрой загрузки, выгрузки или замены данных в секции без длительных массовых операций вставки и удаления.

## 39:00–42:00. Ограничения секционирования

Секционирование имеет ограничения, которые нужно учитывать заранее. Все секции должны иметь тот же набор столбцов, что и секционируемая таблица. Это логично, потому что секции являются частями одной общей таблицы и должны соответствовать её структуре.

Ключ секционирования должен входить в состав первичного ключа или уникального ограничения. Это связано с тем, что СУБД должна гарантировать уникальность не внутри одной секции, а в рамках всей логической таблицы. Если уникальный индекс не включает столбец секционирования, СУБД не всегда может корректно обеспечить глобальную уникальность.

Также уникальность поддерживается только по ключу секционирования. Нельзя создать уникальный индекс по другим полям, если они не включают столбцы секционирования. Ещё одно ограничение: уже существующую таблицу нельзя просто одной командой превратить в декларативно секционированную. Для этого нужна отдельная процедура миграции.

## 42:00–45:00. Особенности секционирования в MySQL

В MySQL секционирование имеет собственные ограничения и особенности. На первом уровне поддерживаются стратегии RANGE или LIST. На втором уровне может использоваться HASH. То есть возможна двухуровневая структура: родительская таблица делится на секции, а секции дополнительно делятся на subpartition.

Глубина такой структуры ограничена двумя уровнями: parent и subpartitions. При этом subpartition может быть только HASH. Это отличается от более гибких моделей в других СУБД.

Практический смысл MySQL-секционирования сохраняется тот же: логически таблица остаётся одной, но физически данные распределяются. Пользователь может не обращаться к секциям напрямую, однако СУБД может использовать это распределение для оптимизации запросов и обслуживания данных.

## 45:00–48:00. Стратегии секционирования

Существует несколько основных стратегий секционирования. RANGE делит данные по диапазонам значений и поддерживается PostgreSQL, Oracle, MySQL и MS SQL Server. LIST распределяет строки по спискам значений и поддерживается PostgreSQL, Oracle и MySQL.

HASH распределяет строки по результату хэширования ключа. Эта стратегия есть в PostgreSQL, Oracle и MySQL. INTERVAL предполагает автоматическое создание секций и характерен для Oracle. KEY — вариант хэширования по ключу, используемый в MySQL.

Выбор стратегии зависит от характера данных и запросов. Если данные естественно делятся по времени, чаще используется RANGE. Если есть устойчивые категории, например регионы или типы клиентов, подходит LIST. Если нужно равномерно распределить нагрузку без смыслового деления данных, используется HASH.

## 48:00–51:00. Partition pruning: исключение секций

Partition pruning — это механизм исключения нерелевантных секций при выполнении запроса. Он позволяет СУБД не читать те секции, в которых заведомо не может быть нужных строк. Именно этот механизм даёт основной прирост производительности от секционирования.

В PostgreSQL, Oracle и MySQL исключение секций обычно происходит на этапе планирования запроса. В MS SQL Server возможна работа как на этапе планирования, так и во время выполнения, что называется runtime elimination.

Для PostgreSQL важно, чтобы был включён параметр:

enable_partition_pruning = on

Но одного факта секционирования недостаточно. Запрос должен содержать фильтр по ключу секционирования, иначе оптимизатор не сможет понять, какие секции можно исключить.

## 51:00–54:00. Когда partition pruning не срабатывает

Partition pruning работает только тогда, когда условие запроса можно напрямую сопоставить с границами секций. Если фильтр вычисляется слишком поздно или ключ секционирования скрыт внутри функции, СУБД может быть вынуждена просматривать больше секций, чем нужно.

Пример неудачного условия:

WHERE date(created_at) = '2024-01-01'

В этом случае функция date() применяется к столбцу, и оптимизатору может быть сложнее использовать границы секций. Более корректный вариант для временного диапазона — писать условие напрямую по исходному столбцу:

WHERE created_at >= '2024-01-01'

AND created_at <  '2024-01-02'

Такая форма лучше соответствует диапазонному секционированию и повышает вероятность исключения лишних секций.

## 54:00–57:00. Секционирование по хэшу

Секционирование по хэшу применяется, когда нужно равномерно распределить строки между несколькими секциями, но нет удобного естественного диапазона или списка значений. Например, данные можно распределять по идентификатору пользователя, заказа или другой стабильной колонке.

В PostgreSQL основная таблица создаётся так:

CREATE TABLE имя_таблицы (

список_столбцов

)

PARTITION BY HASH (ключ_секционирования);

Затем создаются отдельные секции:

CREATE TABLE имя_секции

PARTITION OF имя_таблицы

FOR VALUES WITH (MODULUS N, REMAINDER R);

MODULUS N означает общее количество секций, а REMAINDER R — номер остатка от 0 до N-1. Каждая строка по значению ключа попадает в одну из секций.

## 57:00–60:00. Вопрос о размере таблиц для секционирования

В чате появляется вопрос о том, на каких объёмах нужно делать секционирование таблиц. Формулировка проблемы такая: если есть некоторая грань, после которой индекс уже не спасает, значит таблицу нужно дробить.

Ответ строится на том, что универсального числа строк нет. Нельзя сказать, что секционирование всегда нужно после миллиона, десяти миллионов или ста миллионов строк. Решение зависит от характера запросов, размера строк, объёма чтения, частоты записи, индексов, оборудования и требований к времени ответа.

Секционирование становится оправданным, когда таблица уже велика для обслуживания или когда запросы регулярно читают большие диапазоны данных, которые можно естественно разделить. Например, если почти все запросы выбирают данные за конкретный месяц, а таблица хранит историю за годы, диапазонное секционирование по дате может дать заметный эффект.

## 60:00–63:00. Секционирование по диапазонам RANGE

RANGE-секционирование делит строки по непересекающимся интервалам значений. Каждое значение должно попадать ровно в одну секцию. В PostgreSQL для этого используется форма FOR VALUES FROM ... TO ....

Такая стратегия особенно удобна для временных данных. Например, можно хранить данные за январь и февраль в одной секции, за март и апрель — в другой, за май и июнь — в третьей. Аналогично можно секционировать по числовым диапазонам: идентификаторам, номерам, стоимости или другим упорядоченным значениям.

Ключевое требование — границы диапазонов не должны пересекаться. Если диапазоны заданы неверно, часть данных может не попасть в нужную секцию или возникнут ошибки при вставке. Поэтому RANGE хорошо работает там, где данные имеют естественную последовательность и заранее понятные интервалы.

## 63:00–66:00. RANGE как типовой способ работы с временными данными

Диапазонное секционирование чаще всего применяется к таблицам, где есть дата или время создания записи. Это могут быть журналы событий, история заказов, финансовые операции, логи авторизации, телеметрия и другие данные, которые растут во времени.

Преимущество такого подхода не только в ускорении запросов. Старые секции можно архивировать, отключать, переносить на другое хранилище или удалять целиком. Это намного эффективнее, чем удалять старые строки из одной огромной таблицы через DELETE.

Если запросы почти всегда фильтруются по периоду, например «за день», «за месяц», «за квартал», RANGE-секционирование по дате становится естественным проектным решением. Но если запросы редко используют дату, такой ключ секционирования может не дать ожидаемого результата.

## 66:00–69:00. Когда секционирование действительно полезно

Секционирование полезно не просто потому, что таблица большая. Оно полезно тогда, когда данные можно разделить по признаку, который реально используется в запросах и обслуживании. Если таблица большая, но основные запросы всегда ищут одну строку по первичному ключу и хорошо покрываются индексом, секционирование может не дать заметного выигрыша.

Другая ситуация — таблица большая, но аналитические запросы читают крупные периоды или категории. Тогда секционирование позволяет отсечь ненужные части и ускорить чтение. Особенно это заметно на исторических данных, где активная часть небольшая, а архивная часть занимает большую долю таблицы.

Также секционирование может быть нужно для эксплуатационных задач. Даже если запросы работают приемлемо, обслуживание одной огромной таблицы может быть неудобным: долгий VACUUM, долгий REINDEX, тяжёлые бэкапы, сложное удаление старых данных. В таких случаях секционирование решает не только проблему скорости запросов, но и проблему сопровождения.

## 69:00–72:00. Если таблица уже большая

Если таблица уже большая, её нельзя просто объявить секционированной одной командой. Для существующей таблицы нужен сценарий миграции. Сначала нужно понять, по какому признаку таблица будет делиться, какие диапазоны или группы будут использоваться и как обеспечить корректность данных.

Один из подходов — добавить CHECK-ограничение, соответствующее будущему диапазону секции:

ALTER TABLE имя_таблицы

ADD CONSTRAINT имя_ограничения

CHECK (условие) NOT VALID;

NOT VALID позволяет создать ограничение без немедленной полной проверки всех данных. Это полезно для больших таблиц, потому что полная проверка может быть дорогой операцией.

## 72:00–75:00. Превращение существующей таблицы в секцию

После подготовки ограничения создаётся новая таблица с той же структурой, но уже с PARTITION BY. Затем старая таблица может быть присоединена к новой секционированной таблице как секция.

Примерная форма:

ALTER TABLE новая_табл

ATTACH PARTITION старая_табл

FOR VALUES ...;

После присоединения выполняется валидация ограничения:

ALTER TABLE имя_таблицы

VALIDATE CONSTRAINT имя_ограничения;

Смысл такой процедуры в том, чтобы безопасно включить существующие данные в секционированную структуру. При этом нужно заранее проверить, что строки действительно соответствуют диапазону секции. Иначе присоединение может привести к ошибке или некорректной структуре хранения.

## 75:00–77:00. Итоги занятия

В финале подводятся основные итоги. Секционирование — это способ разбиения большой логической таблицы на физические части. Оно применяется для повышения производительности запросов, уменьшения объёма читаемых данных, упрощения обслуживания, архивирования и резервного копирования.

Главный механизм производительности — partition pruning. Если запрос фильтрует данные по ключу секционирования, СУБД может исключить ненужные секции и читать только релевантную часть таблицы. Если запросы не используют ключ секционирования или используют его в неудачной форме, эффект может быть минимальным.

Основные стратегии секционирования — RANGE, LIST и HASH. RANGE подходит для дат и диапазонов, LIST — для фиксированных категорий, HASH — для равномерного распределения данных. Разные СУБД реализуют эти механизмы по-разному: PostgreSQL использует декларативное секционирование через PARTITION BY, MS SQL Server — partition function и partition scheme, Oracle предоставляет развитые операции управления секциями, а MySQL поддерживает собственную модель с ограниченной глубиной подчинённых секций.
