# Хранимые процедуры в SQL

## Превью / аннотация / краткое содержание

Занятие посвящено хранимым процедурам как объектам базы данных, предназначенным для выполнения действий, бизнес-процессов и многошаговых операций. Центральная проблема связана с разграничением процедур и пользовательских функций, управлением параметрами, обработкой ошибок и безопасным применением динамического SQL. Содержание охватывает синтаксис процедур в PostgreSQL и MS SQL Server, вызов процедур, диагностические сообщения, исключения, транзакционную логику и практические сценарии применения.

## Основная часть

### Хранимая процедура как объект базы данных

Хранимая процедура представляет собой именованный объект базы данных, содержащий последовательность действий, выполняемых по отдельному вызову. В отличие от одиночного SQL-запроса, процедура описывает не только получение результата, но и управляемый процесс обработки данных. Она может выполнять проверки, изменять таблицы, вызывать вспомогательные операции, формировать служебные сообщения и взаимодействовать с транзакционным контекстом. Процедура концептуально связана с бизнес-процессом, поскольку фиксирует не вычисление отдельного значения, а последовательность действий над состоянием базы данных.

#### Назначение хранимых процедур

Хранимые процедуры применяются для централизации операций, которые должны выполняться единообразно независимо от клиентского приложения. Они удобны для многошаговой обработки, модификации данных, проверок существования сущностей, расчётов, логирования и контролируемого завершения операции. Процедура может не возвращать результат как значение, но при этом выполнять существенное действие. При необходимости она способна вернуть несколько наборов данных через SELECT или передать отдельные результаты через OUT-параметры. Такая модель делает процедуру инструментом операционного управления данными.

### Отличие процедур от пользовательских функций

Пользовательская функция ориентирована на вычисление и возврат значения, тогда как хранимая процедура ориентирована на выполнение действия. Функции могут использоваться внутри SELECT, WHERE, JOIN и других частей запроса, поскольку являются частью выражения или источником данных. Процедуры вызываются отдельной командой CALL или EXEC и не встраиваются в обычный SELECT-запрос. Функции в большинстве СУБД ограничены в изменении данных, тогда как процедуры могут выполнять INSERT, UPDATE, DELETE и другие операции модификации. Различие отражает разные роли: функция как вычисление, процедура как процесс.

#### Контекст выполнения функций и процедур

Функция выполняется в контексте вызывающего запроса и подчиняется его логике получения результата. Процедура выполняется независимо как отдельная операция, что позволяет связывать несколько SQL-команд в единый сценарий. Через OUT-параметры процедура может возвращать значения, но это не делает её аналогом функции, поскольку основной смысл процедуры остаётся действием. Процедура может управлять транзакциями, включая BEGIN TRANSACTION, COMMIT и ROLLBACK, если это допускается конкретной СУБД и контекстом вызова. Такое отличие принципиально для проектирования архитектуры базы данных.

### Синтаксис хранимых процедур

Создание процедуры включает имя, список параметров, язык реализации и тело процедуры. В PostgreSQL используется конструкция CREATE OR REPLACE PROCEDURE, позволяющая создать новый объект или заменить существующее определение. В сигнатуре могут задаваться IN, OUT и INOUT-параметры, после чего указывается LANGUAGE plpgsql и процедурный блок BEGIN END. В MS SQL Server применяется синтаксис CREATE PROCEDURE или CREATE OR ALTER PROCEDURE, а вызов выполняется через EXEC. Общая структура сохраняет идею процедурного блока, но конкретные правила зависят от диалекта СУБД.

#### Параметры процедуры

IN-параметры передают значения внутрь процедуры и используются для поиска, проверки или модификации данных. OUT-параметры возвращают значения вызывающей стороне и позволяют сообщить итог операции, статус или вычисленный результат. INOUT-параметры совмещают обе роли: процедура получает исходное значение, изменяет его и возвращает обновлённое состояние. В PostgreSQL процедура может объявлять OUT-параметр в сигнатуре, а при вызове через CALL для него передаётся переменная или специальная позиция результата. Параметры образуют интерфейс процедуры, поэтому их имена, типы и назначение должны быть однозначными.

#### Вызов процедуры

Процедура вызывается не как выражение, а как самостоятельная команда. В PostgreSQL используется CALL, например вызов процедуры с входными значениями и OUT-параметром. В MS SQL Server применяется EXEC или EXECUTE, где параметры могут передаваться позиционно или по имени. Такой способ вызова подчёркивает независимый характер процедуры: она не является частью выборки, а запускает отдельный сценарий выполнения. При проектировании вызова необходимо учитывать, какие параметры являются входными, какие возвращают результат и какие побочные действия выполняет процедура.

### Диагностические сообщения

Диагностические сообщения представляют собой служебный вывод, формируемый СУБД при выполнении процедуры или процедурного блока. Они не являются результатом запроса и не возвращаются как таблица или значение. Их назначение связано с отладкой, наблюдением за ходом выполнения и фиксацией промежуточных состояний. В PostgreSQL для этого используется RAISE NOTICE, в MS SQL Server используется PRINT, в Oracle применяется DBMS_OUTPUT.PUT_LINE. Диагностические сообщения помогают понимать внутренний ход процедуры, но не должны заменять формализованный результат или журналирование.

#### Роль сообщений в отладке процедур

Сообщения позволяют проверить значения переменных, выбранные ветви условий, факт достижения определённого участка кода и итоговые состояния процедуры. Такой механизм особенно полезен при разработке и демонстрации, когда требуется наблюдать выполнение без изменения структуры результата. В производственном коде чрезмерный диагностический вывод может снижать прозрачность и засорять журналы выполнения. Поэтому сообщения должны использоваться ограниченно: для отладки, временной диагностики или контролируемого информирования о ходе сложной операции. Для постоянной фиксации событий предпочтительнее отдельные таблицы логирования.

### Обработка исключений в процедурах

Обработка исключений позволяет управлять ошибками, возникающими во время выполнения процедуры. Общая логика во всех СУБД строится вокруг трёх этапов: выполнение основного кода, перехват ошибки и завершение или освобождение ресурсов. В PL/pgSQL это выражается через секцию EXCEPTION, где отдельные обработчики реагируют на конкретные типы ошибок. Процедура может обработать ожидаемую ошибку локально, вывести диагностическое сообщение, записать событие в лог или пробросить исключение дальше. Такая модель обеспечивает управляемое завершение операции и предотвращает неявное повреждение состояния данных.

#### Генерация пользовательских исключений

В PostgreSQL пользовательская ошибка может быть создана через RAISE EXCEPTION. Простая форма задаёт текст ошибки, а расширенная форма RAISE EXCEPTION USING позволяет указать MESSAGE и ERRCODE. Для пользовательских исключений часто применяется код SQLSTATE P0001, предназначенный для user-defined exception. Генерация исключения используется при нарушении бизнес-условий, например отсутствии студента, курса или невозможности выполнить операцию. Исключение должно не скрывать проблему, а явно фиксировать некорректное состояние и останавливать выполнение до принятия корректирующего решения.

#### Локальная обработка и проброс ошибки

В PL/pgSQL обработчик WHEN позволяет перехватить конкретную ошибку, например division_by_zero, и выполнить локальное действие. Обработчик WHEN OTHERS используется как общий перехватчик для всех ошибок, не описанных отдельно. Через SQLERRM можно получить текст ошибки и использовать его для диагностического сообщения или записи в журнал. Команда RAISE без дополнительных аргументов пробрасывает текущее исключение дальше, сохраняя факт сбоя для внешнего уровня. Такая схема позволяет совместить локальное логирование с сохранением корректной семантики ошибки.

### Транзакционная логика процедур

Хранимые процедуры могут быть связаны с транзакционным управлением, поскольку часто объединяют несколько операций изменения данных. Процедура может выполнять действия, которые должны завершиться как единый бизнес-процесс: проверка условий, изменение таблиц, запись лога и возврат статуса. В некоторых СУБД процедура может явно управлять транзакцией через BEGIN TRANSACTION, COMMIT и ROLLBACK. Возможность такого управления зависит от диалекта, версии СУБД и контекста вызова. Транзакционная логика требует осторожности, поскольку вложенные вызовы и внешние транзакции могут конфликтовать с внутренними командами завершения.

#### Процедура как граница бизнес-операции

Процедура удобна как граница операции, если необходимо обеспечить согласованность нескольких связанных изменений. Например, при расчёте прогресса, записи результата, создании события активности и проверке существования сущностей все действия должны выполняться в контролируемой последовательности. При ошибке процедура должна либо корректно остановиться, либо выполнить компенсирующую обработку. В отличие от разрозненных SQL-команд, процедура задаёт единый сценарий выполнения. Это повышает воспроизводимость поведения, но требует строгого проектирования ошибок, логирования и завершения транзакций.

### Динамический SQL

Динамический SQL представляет собой SQL-код, который формируется и выполняется во время исполнения процедуры, а не фиксируется заранее. Он применяется, когда имя таблицы или столбца передаётся параметром, требуется гибко построить WHERE-условие, выполнить DDL-операцию или сформировать отчёт, зависящий от входных данных. Динамический SQL расширяет выразительность процедур, позволяя строить запросы в зависимости от runtime-условий. В то же время он снижает статическую проверяемость кода, усложняет отладку и требует специальных мер безопасности.

#### Сценарии применения динамического SQL

Динамический SQL оправдан там, где статический запрос не может выразить изменяемую структуру операции. Это касается универсальных процедур, работающих с разными таблицами, отчётов с переменным набором условий, административных операций CREATE TABLE или ALTER TABLE и сценариев, где структура запроса определяется параметрами. Если изменяются только значения фильтров, предпочтительнее использовать обычную параметризацию без динамической сборки всего текста. Динамическая конструкция должна применяться как необходимость, а не как универсальный способ написания SQL-кода.

### Безопасность динамического SQL

Главная угроза динамического SQL связана с SQL-инъекциями, возникающими при небезопасной подстановке пользовательского ввода в текст запроса. Если строка запроса собирается простым объединением фрагментов, входные данные могут изменить структуру SQL-команды. Дополнительные риски связаны с ошибками типов данных, поскольку динамически сформированный текст проверяется только во время выполнения. Ошибки динамического SQL сложнее диагностировать, так как они проявляются в runtime и не всегда видны при создании процедуры. Поэтому динамический SQL должен сопровождаться параметризацией и экранированием идентификаторов.

#### Параметризация в PostgreSQL и MS SQL Server

В PostgreSQL безопасная сборка динамического SQL выполняется через format, где форматтер %I используется для идентификаторов, %L для литералов, а %s для необработанной подстановки. Значения данных следует передавать через EXECUTE ... USING, чтобы они не встраивались напрямую в текст запроса. В MS SQL Server для динамического выполнения используется sp_executesql, позволяющий передавать параметры отдельно от текста команды. Идентификаторы в SQL Server защищаются через QUOTENAME. Такая модель отделяет структуру запроса от значений и снижает риск инъекций.

### Практическая модель процедуры расчёта прогресса

Практический пример связан с процедурой расчёта прогресса студента по курсу. Процедура использует таблицы студентов, курсов, заданий, отправленных решений и журнала активности. Внутри процедуры проверяется существование студента и курса, после чего рассчитываются суммарный набранный балл, максимальный возможный балл и процент выполнения. При отсутствии студента или курса генерируется исключение с диагностическим текстом. Такая процедура иллюстрирует типовой бизнес-процесс: проверка входных данных, расчёт, фиксация результата и управляемая реакция на ошибку.

#### Проверка данных и расчёт результата

Процедура сначала извлекает имя студента по идентификатору и проверяет, что запись существует. Аналогично проверяется наличие курса, поскольку дальнейший расчёт без валидных ссылок не имеет смысла. Затем через агрегирующий запрос определяется фактически набранная сумма баллов и максимальная сумма по заданиям курса. Процент прогресса вычисляется как отношение набранного результата к максимально возможному, при этом необходимо учитывать ситуацию нулевого максимума. Итоговый результат может быть записан, выведен диагностически или возвращён через параметр в зависимости от интерфейса процедуры.

#### Логирование ошибок и исключений

Практический сценарий обработки ошибок включает функцию или процедуру логирования, фиксирующую источник ошибки и текст SQLERRM. При нарушении внешнего ключа или другой ожидаемой ошибке обработчик записывает событие в журнал, после чего исключение может быть проброшено дальше. Это позволяет сохранить информацию о сбое и одновременно не скрывать ошибку от вызывающего уровня. Отдельный обработчик WHEN OTHERS используется как резервный механизм для непредвиденных ошибок. Такая схема показывает связь процедур, исключений и служебного аудита в едином сценарии выполнения.

### Архитектурные границы хранимых процедур

Хранимые процедуры позволяют переносить существенную часть логики в базу данных, но требуют разграничения ответственности между СУБД и приложением. Они оправданы для операций, тесно связанных с данными, транзакциями, проверками целостности и массовыми изменениями. При чрезмерном накоплении процедурной логики база становится «толстой», а поведение системы хуже отслеживается из прикладного кода. Процедуры также зависят от конкретного диалекта SQL, что снижает переносимость. Поэтому процедура должна иметь чёткую задачу, предсказуемые параметры и контролируемые побочные эффекты.

#### Критерии применения процедур

Процедура целесообразна, когда требуется выполнить бизнес-операцию как единый сценарий, объединяющий проверки, изменения, сообщения, логирование и обработку ошибок. Она менее уместна для простых вычислений, которые лучше оформить функцией, или для выборок, которые естественно выражаются представлением или SELECT-запросом. При наличии динамического SQL необходимо отдельно оценивать безопасность, читаемость и сопровождаемость. Качественная процедура уменьшает дублирование и повышает согласованность операций, но не превращает базу данных в непрозрачный слой скрытой бизнес-логики.

## Итоговые выводы

Хранимая процедура является объектом базы данных, предназначенным для выполнения действий и многошаговых бизнес-операций.

Процедуры отличаются от функций способом вызова, возможностью модификации данных, работой с OUT-параметрами и независимым контекстом выполнения.

Диагностические сообщения и обработка исключений обеспечивают наблюдаемость, управляемое завершение и фиксацию ошибок процедурной логики.

Динамический SQL расширяет гибкость процедур, но требует параметризации, защиты идентификаторов и контроля риска SQL-инъекций.

Главный результат освоения занятия состоит в понимании хранимой процедуры как управляемой границы бизнес-процесса внутри SQL-системы.

В центре занятия находится процедура как объект базы данных, который предназначен не столько для вычисления значения, сколько для выполнения действия. Если функция чаще отвечает на вопрос «что вернуть», то процедура отвечает на вопрос «какую операцию выполнить». Поэтому хранимые процедуры особенно важны в задачах, где нужно изменить данные, выполнить несколько связанных SQL-команд, обработать исключение или оформить бизнес-операцию внутри базы.

Главная идея занятия: хранимая процедура позволяет вынести часть прикладной логики на уровень СУБД и запускать её отдельной командой. Это делает работу с данными более централизованной, но одновременно требует внимательного отношения к ошибкам, безопасности, транзакциям и сопровождаемости кода.

## 03:00–06:00. Маршрут вебинара

Структура занятия включает несколько крупных блоков. Сначала рассматривается введение в хранимые процедуры: что такое процедура, зачем она нужна и чем она отличается от пользовательской функции. Далее разбирается синтаксис создания процедур и способы их вызова в разных СУБД.

Отдельный блок посвящён диагностическим сообщениям. Они используются для отладки и позволяют выводить служебную информацию во время выполнения процедуры. В PostgreSQL для этого применяется RAISE NOTICE, в MS SQL Server — PRINT, в Oracle — DBMS_OUTPUT.PUT_LINE.

Следующая часть связана с обработкой исключений. Рассматривается общий принцип try / catch / finally, его реализация в PostgreSQL и MS SQL Server, а также генерация собственных исключений. После этого занятие переходит к динамическому SQL, SQL-инъекциям, параметризации и практической работе в pgAdmin.

## 06:00–09:00. Понятие хранимой процедуры

Хранимая процедура — это именованный объект базы данных, который хранится внутри СУБД и выполняется по отдельному вызову. Процедура может не возвращать результат, а просто выполнять действия: изменять данные, создавать записи, проверять условия, вызывать другие процедуры или фиксировать служебные события.

В отличие от функции, процедура запускается отдельной командой, например CALL в PostgreSQL или EXEC в MS SQL Server. Она не используется как часть выражения внутри SELECT, WHERE или JOIN. Это принципиальное отличие: функция встраивается в запрос, а процедура выполняется как самостоятельная операция.

Процедура может возвращать значения через выходные параметры. Также в некоторых СУБД процедуры могут возвращать несколько наборов данных. Ещё одна важная особенность — возможность управления транзакциями: процедура может быть связана с командами BEGIN TRANSACTION, COMMIT и ROLLBACK, если это разрешено конкретной СУБД и контекстом выполнения.

## 09:00–12:00. Отличие пользовательских функций от хранимых процедур

Пользовательская функция предназначена прежде всего для вычисления и возврата значения. Она может возвращать число, строку, дату, логическое значение, таблицу или другой результат. Функцию можно использовать внутри SQL-запроса, например в SELECT, WHERE, JOIN и других частях выражения.

Хранимая процедура предназначена для выполнения операции. Она может изменять данные, выполнять INSERT, UPDATE, DELETE, вызывать другие процедуры, формировать служебные сообщения и работать как отдельный шаг бизнес-процесса. Возврат значения для процедуры не является обязательным; если результат нужен, он обычно передаётся через OUT-параметры.

Контекст выполнения также различается. Функция выполняется внутри вызывающего SQL-запроса, а процедура запускается независимо. Поэтому функции чаще используются для расчётов и преобразований, а процедуры — для операций, где есть последовательность действий и возможные побочные эффекты.

## 12:00–15:00. Синтаксис процедуры в PostgreSQL

В PostgreSQL процедура создаётся с помощью конструкции CREATE PROCEDURE или CREATE OR REPLACE PROCEDURE. В объявлении указывается имя процедуры, список параметров, язык реализации и тело процедуры. Чаще всего для процедурной логики используется язык plpgsql.

Пример общей формы:

CREATE OR REPLACE PROCEDURE public.process_order(

IN p_order_id int,

IN p_status text,

OUT p_result text

)

LANGUAGE plpgsql

AS $$

BEGIN

-- тело процедуры

END;

$$;

В этом примере процедура принимает идентификатор заказа и статус, а результат возвращает через выходной параметр p_result. Входные параметры используются внутри процедуры для выполнения логики, а выходной параметр заполняется в ходе работы процедуры и передаёт результат вызывающей стороне.

## 15:00–18:00. Вызов процедур в PostgreSQL

В PostgreSQL процедура вызывается командой CALL. Если процедура принимает только входные параметры, вызов выглядит как обычная передача аргументов:

CALL my_procedure(1, 'test');

Если процедура имеет выходной параметр, он также учитывается при вызове. В учебном примере используется форма:

CALL public.process_order(10, 'completed', NULL);

Смысл такого вызова в том, что процедура выполняет действие и возвращает результат не через SELECT, а через специальный параметр. Это подчёркивает отличие процедуры от функции: функция возвращает значение как результат выражения, а процедура выполняется как команда.

## 18:00–21:00. Диагностические сообщения

Диагностические сообщения — это служебные сообщения, которые СУБД выводит во время выполнения процедур или запросов. Они не являются результатом запроса и не возвращаются как данные. Их задача — помочь разработчику увидеть, что происходит внутри процедуры во время выполнения.

В PostgreSQL для таких сообщений используется:

RAISE NOTICE 'Сообщение';

В MS SQL Server аналогичную роль выполняет:

PRINT 'Сообщение';

В Oracle для вывода служебных сообщений применяется:

DBMS_OUTPUT.PUT_LINE('Сообщение');

Такие сообщения особенно полезны при отладке процедурной логики. Через них можно вывести текущие значения переменных, показать этап выполнения процедуры, зафиксировать срабатывание условия или подтвердить, что определённая ветка кода была выполнена.

## 21:00–24:00. Базовая концепция обработки исключений

Исключения — это ошибки, которые возникают во время выполнения кода. Например, деление на ноль, нарушение внешнего ключа, некорректное преобразование типа или попытка обратиться к отсутствующим данным. Если такие ошибки не обрабатывать, выполнение процедуры может завершиться аварийно.

Общая логика обработки исключений во всех СУБД строится на одном принципе. Есть основной блок, где выполняется код. Если в нём возникает ошибка, управление передаётся в блок обработки. После этого можно вывести сообщение, выполнить альтернативное действие, записать ошибку в журнал или пробросить её дальше.

Условно это соответствует схеме try / catch / finally: try выполняет основной код, catch перехватывает ошибку, а finally завершает выполнение или освобождает ресурсы. В разных СУБД синтаксис отличается, но смысл остаётся одинаковым.

## 24:00–27:00. Генерация исключений в PostgreSQL

В PostgreSQL исключение можно создать вручную с помощью RAISE EXCEPTION. Это используется, когда процедура сама обнаруживает недопустимую ситуацию и должна остановить выполнение.

Простой пример:

RAISE EXCEPTION 'Ошибка!';

Более подробный вариант позволяет указать сообщение и код ошибки:

RAISE EXCEPTION USING

MESSAGE = 'Ошибка при переводе',

ERRCODE = 'P0001';

Пользовательские ошибки в PostgreSQL часто задаются с кодом P0001, который относится к user-defined exception. Также можно указывать другие коды из допустимых диапазонов, если они не конфликтуют со встроенными ошибками СУБД.

## 27:00–30:00. Обработка исключений в PostgreSQL

В PostgreSQL обработка исключений реализуется внутри блока BEGIN ... EXCEPTION ... END. Сначала выполняется основной код, затем при возникновении ошибки управление переходит к соответствующему обработчику.

Пример:

DO $$

BEGIN

PERFORM 10 / 0;

EXCEPTION

WHEN division_by_zero THEN

RAISE NOTICE 'Ошибка: деление на ноль. Обработали локально.';

WHEN OTHERS THEN

RAISE NOTICE 'Другая ошибка: %', SQLERRM;

RAISE;

END;

$$;

Здесь ошибка деления на ноль обрабатывается отдельно. Для всех остальных ошибок используется WHEN OTHERS. Переменная SQLERRM позволяет получить текст ошибки. Команда RAISE без аргументов пробрасывает ошибку дальше, если её нельзя полностью обработать на текущем уровне.

## 30:00–33:00. Обработка исключений в MS SQL Server

В MS SQL Server используется конструкция BEGIN TRY ... END TRY и BEGIN CATCH ... END CATCH. В блоке TRY выполняется основной код, а в блоке CATCH описывается реакция на ошибку.

Пример:

BEGIN TRY

SELECT 10 / 0;

END TRY

BEGIN CATCH

IF ERROR_NUMBER() = 8134

PRINT 'Ошибка: деление на ноль. Обработали локально.';

ELSE

BEGIN

PRINT 'Другая ошибка: ' + ERROR_MESSAGE();

THROW;

END

END CATCH;

Функция ERROR_NUMBER() позволяет определить номер ошибки, а ERROR_MESSAGE() возвращает текст сообщения. Для деления на ноль используется код 8134. Если ошибка не относится к ожидаемым, её можно пробросить дальше с помощью THROW.

## 33:00–36:00. Переход к динамическому SQL

После обработки исключений рассматривается динамический SQL. Это SQL-код, который формируется не заранее, а во время выполнения процедуры. То есть процедура не содержит полностью готовый запрос, а собирает его из отдельных частей: имени таблицы, имени колонки, условий фильтрации, сортировки или других параметров.

Динамический SQL используется, когда обычный статический запрос не подходит. Например, если имя таблицы или колонки передаётся параметром, если нужно построить гибкий WHERE, если выполняются DDL-операции вроде CREATE TABLE или ALTER TABLE, либо если строится отчёт, зависящий от времени или входных данных.

Главная особенность динамического SQL в том, что запрос становится строкой. Это даёт гибкость, но одновременно снижает безопасность и усложняет отладку, потому что ошибка может проявиться только в момент выполнения сформированного запроса.

## 36:00–39:00. Опасности динамического SQL

Основная опасность динамического SQL — SQL-инъекции. Они возникают, когда пользовательское значение напрямую вставляется в текст SQL-запроса. В таком случае введённая строка может быть воспринята не как данные, а как часть исполняемой команды.

Кроме SQL-инъекций есть и другие проблемы. Поскольку в динамическом SQL всё часто формируется как строка, легко ошибиться с типами данных. Например, число может быть передано как текст, дата может быть неправильно заключена в кавычки, а идентификатор может быть вставлен без экранирования.

Отладка динамического SQL также сложнее. Статический запрос проверяется заранее, а динамический формируется во время выполнения. Поэтому синтаксическая ошибка, ошибка имени таблицы или ошибка приведения типа может появиться только в runtime. Из-за этого динамический SQL нужно применять осознанно и только там, где он действительно необходим.

## 39:00–42:00. Параметризация в PostgreSQL

Для безопасного динамического SQL в PostgreSQL используется параметризация. Текст запроса формируется отдельно, а значения передаются через USING. Это позволяет СУБД воспринимать значения именно как данные, а не как часть SQL-кода.

Пример:

sql := format(

'SELECT * FROM %I WHERE created_at > $1',

p_table

);

EXECUTE sql USING p_date;

Здесь имя таблицы подставляется через format и %I, то есть как идентификатор. Дата передаётся через $1 и USING, поэтому она не встраивается в строку запроса напрямую. Это защищает от SQL-инъекции для значений.

В PostgreSQL используются разные форматтеры: %I — для идентификаторов, например имён таблиц и колонок; %L — для литералов, то есть строк в кавычках; %s — для необработанной подстановки. Последний вариант требует осторожности, потому что не выполняет защитного экранирования.

## 42:00–45:00. Параметризация в MS SQL Server

В MS SQL Server безопасное выполнение динамического SQL обычно строится через sp_executesql. Запрос формируется как строка, но значения параметров передаются отдельно.

Пример:

DECLARE @sql nvarchar(max);

SET @sql = N'

SELECT *

FROM ' + QUOTENAME(@table) + '

WHERE created_at > @p_date';

EXEC sp_executesql

@sql,

N'@p_date datetime',

@p_date = @date;

Здесь QUOTENAME используется для безопасной обработки имени таблицы, а значение даты передаётся как параметр. Такая схема принципиально отличается от простой конкатенации строки, потому что данные не превращаются в исполняемый SQL-код.

Главная идея параметризации одинакова для разных СУБД: структура запроса может формироваться динамически, но значения должны передаваться через параметры. Это снижает риск инъекций и делает выполнение запроса более предсказуемым.

## 45:00–48:00. Переход к практической части

После теоретических блоков занятие переходит в pgAdmin. Практика выполняется на учебной базе данных, где уже есть таблицы студентов, курсов, заданий, отправленных решений и журнала активности.

Используются таблицы students, courses, assignments, submissions, enrollments и activity_log. Такая схема позволяет показать хранимые процедуры на более реалистичном примере: нужно проверить существование студента и курса, посчитать прогресс, обработать ошибки, записать действия в журнал и вывести диагностические сообщения.

Практическая часть демонстрирует, что процедура может объединять несколько задач: проверку входных параметров, выборку данных, расчёт результата, обработку исключений и логирование.

## 48:00–51:00. Простая процедура расчёта прогресса студента

Первый практический пример — процедура student_course_progress. Она рассчитывает прогресс студента по конкретному курсу. Процедура принимает идентификатор студента и идентификатор курса.

Внутри процедуры объявляются переменные:

v_student_name text;

v_course_title text;

v_total_score int;

v_max_score int;

v_percent numeric(5,2);

Префикс v_ используется для переменных. Это не обязательное требование SQL, а соглашение о стиле, которое помогает отличать локальные переменные от параметров процедуры и от имён колонок. Параметры в примерах обозначаются через p_, например p_student_id и p_course_id.

## 51:00–54:00. Проверка существования студента и курса

В начале процедуры выполняется проверка, что студент существует. Для этого из таблицы students выбирается имя студента по p_student_id. Если запись не найдена, процедура генерирует исключение.

Пример логики:

SELECT s.full_name

INTO v_student_name

FROM students s

WHERE s.student_id = p_student_id;

IF v_student_name IS NULL THEN

RAISE EXCEPTION 'Студент с id = % не найден', p_student_id;

END IF;

Аналогично проверяется существование курса. Из таблицы courses выбирается название курса, и если курс не найден, выполнение процедуры прерывается через RAISE EXCEPTION. Это показывает базовую валидацию входных параметров внутри процедуры.

## 54:00–57:00. Расчёт баллов по курсу

После проверок процедура рассчитывает суммарный балл студента и максимально возможный балл по курсу. Для этого используется соединение таблиц assignments и submissions.

Логика расчёта строится так: все задания курса дают максимальную сумму баллов, а из отправленных решений студента учитываются только проверенные работы, то есть записи со статусом graded. Если у студента нет проверенной отправки, балл не должен случайно учитываться как полноценный результат.

Для суммирования используется coalesce, чтобы заменить NULL на ноль. Это важно, потому что агрегатные функции могут возвращать NULL, если подходящих строк нет. В результате процедура получает два значения: фактически набранный балл и максимальный балл по курсу.

## 57:00–60:00. Обработка случая отсутствия проверенных заданий

Если максимальный балл по курсу равен нулю, нельзя выполнять деление при расчёте процента. Поэтому в процедуре отдельно обрабатывается ситуация, когда по курсу нет проверенных заданий или нет данных для расчёта.

В таком случае процедура выводит диагностическое сообщение через RAISE NOTICE. Сообщение содержит имя студента и название курса, чтобы было понятно, к какой ситуации относится результат.

Затем процедура завершает выполнение через RETURN. Это пример контролируемого выхода из процедуры: ошибка не обязательно должна приводить к аварийному завершению, если ситуация ожидаемая и её можно обработать штатно.

## 60:00–63:00. Журналирование ошибок

Следующий практический блок связан с обработкой исключений и журналированием. Создаётся вспомогательная логика, которая записывает сведения об ошибке в отдельную таблицу. Это позволяет не просто вывести сообщение пользователю, а сохранить техническую информацию для последующего анализа.

В журнал может записываться имя процедуры, текст ошибки, контекст выполнения и дополнительные детали. Такой подход полезен в реальных системах, потому что пользователь может увидеть только общее сообщение, а разработчику нужна точная причина сбоя.

В примере используется процедура или функция логирования вида log_error, куда передаются название места возникновения ошибки и SQLERRM, то есть текст ошибки PostgreSQL.

## 63:00–66:00. Обработка конкретной ошибки внешнего ключа

Демонстрируется ситуация, когда процедура специально пытается вставить запись с несуществующим студентом. Например, в таблицу enrollments добавляется student_id = 9999, которого нет в таблице students.

Такая операция нарушает внешний ключ и вызывает ошибку foreign_key_violation. В блоке EXCEPTION эта ошибка перехватывается отдельно:

EXCEPTION

WHEN foreign_key_violation THEN

PERFORM log_error(

'demo_exception_handling/enrollments',

SQLERRM

);

RAISE;

После записи ошибки в журнал исключение пробрасывается дальше. Это важный принцип: если процедура не может полностью исправить проблему, она не должна скрывать ошибку. Она может зафиксировать её, но затем должна дать вызывающей стороне понять, что операция не выполнена.

## 66:00–69:00. Общий обработчик исключений

Помимо конкретной ошибки внешнего ключа используется общий обработчик WHEN OTHERS. Он срабатывает для всех ошибок, которые не были обработаны отдельными ветками.

Пример:

WHEN OTHERS THEN

PERFORM log_error(

'demo_exception_handling/other',

SQLERRM

);

RAISE;

Такой обработчик нужен как страховка, но использовать его нужно осторожно. Если просто перехватывать все ошибки и ничего не делать, можно скрыть серьёзные проблемы. Поэтому в учебном примере ошибка сначала записывается в лог, а затем снова пробрасывается через RAISE.

Отдельно показывается, что код после RAISE уже не будет выполнен. Если исключение проброшено дальше, управление выходит из текущего блока.

## 69:00–72:00. Контекст вложенных процедур

Следующий пример показывает цепочку вложенных процедур. Есть внутренняя процедура, которая генерирует ошибку, промежуточная процедура-обёртка и внешняя процедура, которая запускает весь процесс.

Примерная логика такая: outer_procedure_entry вызывает middle_procedure_wrapper, а та вызывает inner_procedure_fail. Ошибка возникает во внутренней процедуре, но в сообщении PostgreSQL можно увидеть весь контекст вызовов.

Контекст помогает понять, где именно возникла проблема и через какую цепочку вызовов она прошла. Это особенно важно в сложных системах, где одна процедура вызывает другую, а ошибка может проявиться только на верхнем уровне.

## 72:00–75:00. Демонстрация SQL-инъекции

После блока исключений практика переходит к динамическому SQL. Создаётся демонстрационная таблица sql_injection_demo, а затем уязвимая процедура, которая принимает строку и напрямую подставляет её в SQL-запрос.

Уязвимая логика выглядит по смыслу так:

v_sql := 'insert into sql_injection_demo(id) values (' || p_raw || ');';

EXECUTE v_sql;

Если передать обычное значение, например 1, процедура выполнит вставку. Но если передать строку с дополнительной SQL-командой, например с drop table, эта команда тоже может попасть в исполняемый запрос. Так показывается, почему простая конкатенация пользовательского ввода опасна.

## 75:00–78:00. Безопасная версия динамического SQL

После уязвимого примера создаётся безопасная процедура. Она принимает значение как параметр определённого типа, например int, и выполняет динамический SQL через USING.

Пример:

CREATE OR REPLACE PROCEDURE safe_insert_demo(p_value int)

LANGUAGE plpgsql

AS $$

BEGIN

RAISE NOTICE 'Исполняем динамический SQL (безопасная версия)';

EXECUTE 'insert into sql_injection_demo(id) values ($1)'

USING p_value;

END;

$$;

В этом случае значение не встраивается в строку запроса. Оно передаётся как параметр, поэтому PostgreSQL воспринимает его как данные. Если попытаться передать строку с SQL-инъекцией туда, где ожидается целое число, возникнет ошибка приведения типа, а не выполнение вредоносной команды.

## 78:00–81:00. Динамический отчёт и проверка параметров

Далее рассматривается более прикладной пример динамического SQL: процедура формирует отчёт по студентам и их суммарным баллам по курсу. В запросе используются таблицы students, enrollments, assignments и submissions.

Процедура строит SQL-запрос через format, подставляя условие сортировки. Перед этим проверяется параметр сортировки, например name или score_desc. Если передано недопустимое значение, процедура генерирует исключение.

Это показывает важный принцип: если часть структуры SQL-запроса задаётся параметром, её нужно ограничивать заранее известным набором допустимых значений. Нельзя напрямую принимать от пользователя произвольный фрагмент ORDER BY, потому что это снова создаёт риск SQL-инъекции.

## 81:00–84:00. Управление транзакциями

Следующий блок посвящён транзакциям. Транзакция задаёт границы атомарного выполнения: либо все операции внутри блока фиксируются, либо все откатываются. Для процедур это особенно важно, потому что одна процедура может выполнять несколько связанных изменений данных.

В PostgreSQL демонстрируется попытка управлять транзакцией внутри функции. Создаётся пример, где внутри функции выполняется START TRANSACTION, затем UPDATE, затем COMMIT. При запуске PostgreSQL возвращает ошибку, потому что управление транзакциями внутри функции запрещено.

Сообщение ошибки указывает, что команда транзакции не поддерживается в контексте PL/pgSQL-функции. Это демонстрирует различие между функциями и процедурами: транзакционное управление относится именно к процедурам, но даже для процедур оно зависит от контекста вызова.

## 84:00–87:00. Почему транзакциями нельзя управлять в функциях

Функция выполняется внутри вызывающего SQL-запроса. Она не является самостоятельным транзакционным блоком, поэтому не может произвольно выполнять COMMIT или ROLLBACK. Если бы функция могла завершать транзакцию внутри запроса, это нарушило бы предсказуемость выполнения SQL.

Процедура, напротив, рассматривается как самостоятельная операция. Поэтому в PostgreSQL процедуры могут использоваться для транзакционной логики, если вызываются в подходящем контексте. Это ещё раз подчёркивает концептуальное различие: функция — для вычисления и возврата результата, процедура — для действия и управления процессом.

В практическом выводе делается акцент на том, что нельзя механически переносить процедурную логику из функций в процедуры и обратно. Нужно учитывать ограничения конкретного объекта базы данных.

## 87:00–90:00. Модульность процедурной логики

К концу практики рассматривается более крупная процедура пересчёта прогресса по курсу. Она выполняет много действий: проверяет параметры, удаляет старые данные, пересчитывает показатели, группирует студентов по состоянию прогресса и записывает сведения в журнал.

На этом примере показывается проблема больших процедур. Если вся логика находится в одном длинном блоке, код становится трудно читать и сопровождать. Поэтому более правильный подход — делить процедуру на смысловые части: проверка входных данных, расчёт, обновление, логирование, обработка ошибок.

В примере появляется модульная версия процедуры, например recalc_course_progress_modular. Она должна быть более читаемой по сравнению с большой монолитной процедурой.

## 90:00–93:00. Журналирование результата пересчёта

В модульной процедуре после пересчёта прогресса формируются итоговые показатели. Например, количество удалённых снимков прогресса, количество студентов без заданий, без прогресса, нуждающихся в помощи, с нормальным результатом и с отличным результатом.

Если включён параметр логирования, процедура записывает в activity_log JSON-объект с деталями пересчёта:

jsonb_build_object(

'course_id', p_course_id,

'course_title', v_course_title,

'deleted_snapshots', v_deleted,

'no_assignments', v_cnt_no_assignments,

'no_progress', v_cnt_no_progress,

'needs_help', v_cnt_needs_help,

'ok', v_cnt_ok,

'excellent', v_cnt_excellent

)

Такой пример показывает, что процедура может не только выполнять изменение данных, но и сохранять подробный технический итог своей работы. Это полезно для аудита, диагностики и контроля выполнения фоновых операций.

## 93:00–96:13. Завершение занятия

В конце занятия подводится итог по основным темам. Хранимые процедуры используются для выполнения действий внутри базы данных: изменения данных, запуска связанных операций, обработки ошибок, формирования диагностических сообщений и организации серверной логики.

Функции и процедуры имеют разное назначение. Функции возвращают значения и встраиваются в SQL-запросы, а процедуры выполняются отдельной командой и могут представлять собой бизнес-операцию. Для вызова процедур в PostgreSQL используется CALL, в MS SQL Server — EXEC.

Отдельное внимание уделяется безопасности. Динамический SQL нельзя строить через простую конкатенацию пользовательских значений. Для безопасного выполнения нужно использовать параметризацию, EXECUTE ... USING, format, корректное экранирование идентификаторов и проверку допустимых значений.
